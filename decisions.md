# decisions.md: Ledgr Architecture Decision Log

A running log of decisions made *during the build*. The PRD's §10 decisions log is frozen product intent; this file captures the implementation choices that come up as code gets written (library picks, schema judgment calls, trade-offs Claude Code and Brandon settle in the moment).

**Why a separate log:** so the reasoning behind a choice survives past the session that made it. When future-Brandon (or Claude Code) wonders "why did we do it this way," the answer is here, not lost in chat history.

## How to use this
- Add an entry when you make a real architectural or tooling choice, not for routine code.
- Keep entries short: context, the decision, and why (plus what was rejected, if it matters).
- Never rewrite history. If a decision is reversed, add a new entry that supersedes the old one and note it.
- Number sequentially. Date each entry.

### Template
```
## ADR-NNN: <short title>
**Date:** YYYY-MM-DD
**Status:** accepted | superseded by ADR-NNN | reversed
**Context:** what prompted the decision.
**Decision:** what we chose.
**Why / alternatives:** the reasoning; what was rejected and why.
**Affects:** files/areas touched.
```

## ADR-002: Repo scaffold choices
**Date:** 2026-06-12
**Status:** accepted
**Context:** First code slice. create-next-app defaults plus the project's non-negotiables needed concrete picks.
**Decision:**
- Next.js 16 (App Router, TypeScript, Turbopack) scaffolded with create-next-app at `C:\dev\ledgr`. Code lives outside OneDrive on purpose: node_modules and `.next` in a synced folder cause churn, file locks, and slow installs. The project docs are copied into the repo root so GitHub backs them up and Claude Code sessions in the repo have them.
- Tailwind CSS v4 kept from the scaffold default: one well-trodden dependency instead of a bespoke CSS system, consistent with the boring-stack rule.
- Drizzle ORM over the Neon serverless HTTP driver (`drizzle-orm/neon-http`). `src/db/index.ts` is a lazy singleton that refuses any `*.neon.tech` connection string whose host lacks `-pooler`, turning the pooler rule from a convention into a runtime guard. Local Postgres (Phase 4) is exempt from the guard.
- Clerk wired behind a thin `AuthProvider` interface (`src/lib/auth/`): server code calls `authProvider.getCurrentUser()`, the layout wraps `AppAuthProvider`, and both fall back cleanly when no Clerk key is configured. This is the Phase 4 local-mode seam made real, and it also lets the repo build on a fresh clone with no secrets.
- Clerk middleware lives in `src/proxy.ts` (the Next 16 convention replacing `middleware.ts`) and excludes `/health`, since machine endpoints authenticate with scoped API tokens, never Clerk.
- Dependencies pinned exactly (`--save-exact`) per the "updates batched intentionally, never auto" rule.
- `/health` reports DB reachability plus a placeholder `lastExportAt`; raw error detail is gated behind `DEBUG_MODE` so failures stay legible without leaking connection details.
**Why / alternatives:** The Neon HTTP driver was chosen over node-postgres/TCP because it is built for serverless functions and pairs naturally with the pooled endpoint. Requiring Clerk keys at build time was rejected; the no-key fallback keeps the auth seam genuine rather than decorative. Skipping Tailwind was rejected as trading one dependency for a pile of hand-rolled CSS.
**Affects:** `package.json`, `src/db/`, `src/lib/auth/`, `src/proxy.ts`, `src/app/health/`, `drizzle.config.ts`, `.env.example`, runbook §1.

## ADR-003: Schema implementation judgment calls
**Date:** 2026-06-12
**Status:** accepted
**Context:** Implementing schema.md in Drizzle (slice 2) surfaced the deferred decisions schema.md flagged, plus a few implementable-form details.
**Decision:**
- **Entity `kind` is a real column on `items`** (`text`, nullable), not a key in `properties`. It is a hot, filterable field ("show all people"), and hot fields are columns. Plain `text` rather than a Postgres enum so adding a kind never needs a migration.
- **FTS uses an app-maintained `body_text` column** plus a `GENERATED ALWAYS AS (...) STORED` `tsvector` over `title + body_text`, GIN-indexed. Generating the tsvector straight from the BlockNote JSONB was rejected: `body::text` would index structural noise (`"type":"paragraph"`, etc.) and pollute results. App code extracts plain text from the BlockNote doc on save (same save path that snapshots revisions).
- **Child tables hard-cascade:** `relations`, `attachments`, `revisions` have `ON DELETE CASCADE` to `items`, so the 30-day Trash purge (the only hard delete) cleans up edges and snapshots in one statement. Soft-delete cascade to child *items* stays app-level (slice 4).
- **`users.email` is unique and `clerk_id` is nullable-unique**, so the seed is idempotent (`ON CONFLICT DO NOTHING` on email) and the row can exist before Clerk is wired in slice 3.
- **Migrations:** `drizzle-kit generate` produces SQL into `drizzle/`; `scripts/migrate.mjs` applies it via `drizzle-orm/neon-http/migrator`, and `scripts/seed.mjs` seeds via the same Neon HTTP driver. Both are plain Node (`--env-file-if-exists`), reuse existing deps, and enforce the pooler guard. No `tsx`/`dotenv` added (boring-stack rule).
- `matchers` (Phase 2) deliberately not created.
**Why / alternatives:** Each call follows an existing principle (hot-fields-are-columns, no-new-deps, purge must not strand rows). A separate `entities` table for `kind` was rejected (violates everything-is-an-item).
**Affects:** `src/db/schema.ts`, `drizzle/0000_*.sql`, `scripts/migrate.mjs`, `scripts/seed.mjs`, `package.json` (db:* scripts), schema.md corrected per its own header rule.

## ADR-004: Machine API tokens as hashed env entries, no DB table
**Date:** 2026-06-12
**Status:** accepted
**Context:** Slice 3 needs the machine-to-machine auth scheme (MCP, cron, webhooks) the PRD requires to be scoped, revocable, and separate from Clerk. The obvious designs were a `api_tokens` DB table or an env-var scheme.
**Decision:** Tokens live as comma-separated `name:scope1+scope2:sha256hex` entries in the `LEDGR_API_TOKENS` env var. `scripts/make-token.mjs` generates a raw token (`lgr_` + 48 hex chars, shown once, held only by the caller) and its env entry. `src/lib/auth/machine.ts` verifies `Authorization: Bearer` headers by SHA-256 + constant-time compare and returns `{name, scopes}`. Machine routes live under `/api/machine/*`, are excluded from Clerk protection in `src/proxy.ts`, and verify their own token per-handler; `/api/machine/ping` is the diagnostic route (any valid token; echoes the caller's own identity). Revocation = remove the entry, redeploy.
**Why / alternatives:** Single user with a handful of long-lived tokens does not justify a table, issuance UI, or query on every request (fast-and-cheap rule); env entries are zero-dependency and the hash-only storage means neither a DB dump nor an env dump leaks a credential. A DB table becomes the right answer if tokens ever need per-token expiry, usage audit, or self-serve issuance; nothing in the route contract would change, only `verifyMachineToken`'s lookup.
**Affects:** `src/lib/auth/machine.ts`, `src/app/api/machine/ping/`, `src/proxy.ts`, `scripts/make-token.mjs`, `.env.example`, runbook §1/§3.

## ADR-005: Trash/restore semantics, purge-cron auth, and a dev-only TS runner
**Date:** 2026-06-12
**Status:** accepted
**Context:** Slice 4 (item CRUD) had to make the PRD's soft-delete promises concrete: "soft-deleting a parent takes its children to Trash with it so the unit restores together" (§3.5/§4.6), a 30-day purge, and debounced revision snapshots. It also needed the first real scheduled job and a way to verify app code outside an HTTP request.
**Decision:**
- **Deletion units via shared `deleted_at`.** Cascade soft-delete stamps the item and every live descendant with the same `deleted_at` in one recursive-CTE UPDATE. Restore matches on that shared timestamp, so the unit round-trips together while a child trashed earlier in a *separate* delete keeps its own timestamp and stays in Trash. No extra `deletion_group` column needed; the timestamp already encodes the group.
- **Purge is the only hard delete** (`purgeExpiredTrash`): items with `deleted_at` older than 30 days are deleted (child tables cascade per ADR-003) after a detach UPDATE clears `parent_id` on any non-expiring item still pointing at an expiring one (the restored-out-of-a-unit edge case). Two statements without a transaction is accepted: a stray detach is harmless and the delete retries next run.
- **Vercel cron authenticates as a machine token.** `vercel.json` schedules `GET /api/machine/purge` daily (08:00 UTC); Vercel sends `Authorization: Bearer $CRON_SECRET`, and `CRON_SECRET` holds a raw `cron`-scoped token from the ADR-004 scheme. The platform cron walks through the same door as GitHub Actions or any future local scheduler — no second auth path, which keeps the provider-interface discipline (CLAUDE.md) intact. Purge failures insert into `error_log` with a correlation id (no silent failures).
- **Cycle guard at write time.** `parent_id` updates reject self and descendants (recursive CTE check), since one cycle would hang every recursive tree read; the cascade CTEs additionally use `UNION` (not `UNION ALL`) so even bad data can't recurse forever. The fuller subtasks slice still owns recursive reads + rollup.
- **`tsx` added as a devDependency** to run `scripts/verify-items.mts`, which exercises the real `src/lib/items.ts` against Neon. ADR-003 declined `tsx` when plain-Node `.mjs` scripts sufficed; this slice is the first whose verification must import app TypeScript (with `@/` aliases). Dev-only, zero runtime footprint, and it's the seam a future test runner would use. Duplicating the logic in `.mjs` was rejected as verifying a copy, not the code.
**Why / alternatives:** A `deletion_group_id` column was rejected as a second source of truth for what the timestamp already says. Vercel's implicit `CRON_SECRET` comparison alone (without the token scheme) was rejected because it would create an unscoped, unnamed credential outside the rotation runbook. Restoring a child out of a still-trashed parent intentionally keeps `parent_id` (the Trash view shows everything, and restoring the parent later reunites the tree); detach-on-restore was rejected as destroying hierarchy.
**Affects:** `src/lib/items.ts`, `src/lib/body-text.ts`, `src/lib/api.ts`, `src/app/api/items*`, `src/app/api/machine/purge/`, `vercel.json`, `scripts/verify-items.mts`, `package.json` (tsx), runbook §1/§2a/§3.

## ADR-006: Editor slice judgment calls (serializer, storage, mentions, dev auth stand-in)
**Date:** 2026-06-12
**Status:** accepted; **body-format parts revised by ADR-037** (Markdown is now canonical, so the BlockNote→markdown serializer's role inverts — markdown is the stored source and other formats render *from* it). The storage (`aws4fetch`/`StorageProvider`), mention-sync, and dev-auth-stand-in decisions still stand.
**Context:** Slice 5 (BlockNote editor + markdown serialization + image paste + @-mentions) forced four implementation choices the PRD leaves open.
**Decision:**
- **The markdown serializer is hand-rolled, pure JSON-walking code** (`src/lib/markdown.ts`) with no `@blocknote` import, fed by the single color mapping table in `src/lib/colors.ts` (PRD §4.1). BlockNote's own `blocksToMarkdownLossy` was rejected: it drops colors/highlights (the whole point of §4.1's encoding) and would drag the editor bundle into the OneDrive-export cron and Pulpit Ready paths, which must stay server-side and editor-free. The color table pins BlockNote's default palette hexes so an editor upgrade can't silently change what exported documents mean. Highlights emit both the class (`hl-*`, the stable theming hook) and an inline `background-color` style so the exact color renders even with no CSS snippet; mentions export as `[@Title](ledgr://item/<uuid>)`, a stable URI for the future importer/link-rewriter.
- **R2 via `aws4fetch`** (~6KB, zero deps) behind the `StorageProvider` interface (`src/lib/storage/`): presigned PUT from the browser, bytes never proxy through the app. The full AWS SDK was rejected on Principle 5. The attachment **metadata row is created at presign time**, before the upload: an orphaned row from an abandoned upload is harmless, an untracked R2 object would leak quota. Per-file cap 100MB; per-user quota ~10GB enforced by summing `attachments.size_bytes`. Keys are `ownerId/attachmentId/filename` so per-user accounting and cleanup are prefix operations.
- **Mentions sync as `relations` rows with role `'mention'`**, diffed on every body save (create, update, revision restore): rows appear when a mention is added, disappear when it's removed, and rows with any other role are never touched, so the auto-sync can never destroy a manual link. `'mention'` rather than the schema example `'references'` precisely because the sync deletes what it owns; an auto-managed role must not be shared with hand-made edges. Targets are owner-scoped on insert; self-mentions and dangling ids are dropped silently.
- **A dev-only auth stand-in** joins the provider seam (`src/lib/auth/index.ts`): with no Clerk key, `NODE_ENV=development`, and `DEV_USER_EMAIL` set, the app resolves that email as the signed-in user. This is the Phase 4 local single-user mode in miniature and exists so editor UI (and every future UI slice) can be exercised locally without a Microsoft sign-in. Alongside it, **`resolveOwner`'s email backfill now only fills an empty `clerk_id`**, never overwrites an existing link; the old behavior would have let any second provider identity with a matching email silently steal the row.
**Why / alternatives:** Each choice keeps a non-negotiable intact: editor-free exports (rule 3/8), no new heavyweight deps (rule 5), deterministic mention plumbing with no model in the loop (rule 3), and a genuine auth seam (provider-interface discipline). A `mentions` join table separate from `relations` was rejected (relations *is* the unified edge table).
**Affects:** `src/lib/colors.ts`, `src/lib/markdown.ts`, `src/lib/storage/`, `src/lib/attachments.ts`, `src/lib/mentions.ts`, `src/lib/items.ts`, `src/lib/owner.ts`, `src/lib/auth/index.ts`, `src/app/api/attachments/`, `src/components/editor/`, `src/app/items/[id]/`, `package.json` (@blocknote/*, aws4fetch), `.env.example`, runbook §1, `scripts/verify-editor.mts`.

## ADR-007: Item canvas via intercepting routes; hardcoded field-zone table
**Date:** 2026-06-12
**Status:** accepted; **the "every item opens the same editor canvas" premise is revised by ADR-037** — a content type may now declare its own canvas, with the markdown editor as the default. The intercepting-route mechanics, edit-loss seams, and field-zone table still stand for the default canvas.
**Context:** Slice 8 (item canvas, PRD §4.13) needs the center modal as the default open with the URL still routable, a full-screen expand, and per-type field zones, all without forking the editing core.
**Decision:**
- **The modal is a Next.js intercepting + parallel route** (`src/app/@modal/(.)items/[id]` rendered through a `modal` slot in the root layout). Client-side navigation to any `/items/[id]` link intercepts into a center modal over the launching list; a document load (refresh, deep link, the ⤢ Expand control) renders the full page at `src/app/items/[id]`. Expand is therefore a plain `<a>`, not `<Link>`: a soft navigation to the same URL would stay intercepted. Both forms render one shared `ItemCanvas` server component; ItemEditor is untouched except a `fields` slot.
- **Edit-loss seams closed at unmount:** ItemEditor flushes pending autosave edits in an unmount cleanup (keepalive PATCH), since closing the modal inside the 1.5s debounce window would otherwise drop the last edit; the Modal calls `router.refresh()` on unmount so the list underneath shows modal-made changes the moment it closes.
- **The top strip is a hardcoded per-type table** (`src/lib/canvas-fields.ts`): task status/due/urgency, meeting when/status, link url/status, entity kind/status, note status, unknown types status only. PRD §4.13's user-configurable field placement is a Build-surface feature; this table is the seam it replaces. Non-strip fields render read-only in a collapsed Fields footer.
- **Status/urgency enums moved to `src/lib/item-enums.ts`** (re-exported from `items.ts`) because the client-side strip needs the value lists and importing `items.ts` would drag the DB layer into the browser bundle.
- **Esc closes the modal only when unclaimed** (`!e.defaultPrevented`), so BlockNote popovers (slash menu, mention picker) keep their own Esc and the modal closes on the next one (verified in-browser).
**Why / alternatives:** Client-managed modal state (open-in-place without a route) was rejected: the URL wouldn't be routable, breaking refresh, sharing, and the Notion-default contract. A separate modal editor component was rejected as forking the editing core the PRD says carries over unchanged. Per-user opening-mode preference (side panel, new tab) is deferred; the renderer is shared, so adding modes later is chrome, not a rewrite.
**Affects:** `src/app/layout.tsx`, `src/app/@modal/`, `src/app/items/[id]/page.tsx`, `src/components/canvas/` (ItemCanvas, Modal, FieldStrip), `src/components/editor/ItemEditor.tsx`, `src/lib/canvas-fields.ts`, `src/lib/item-enums.ts`, `src/lib/items.ts`.

## ADR-008: "Today" is computed against an explicit app timezone; due dates and meeting times use different day boundaries
**Date:** 2026-06-12
**Status:** accepted
**Context:** The Today view (slice 9) needs day boundaries, but the server runs in UTC on Vercel and the PRD never pins a timezone. Worse, the two date fields encode time differently: `meeting_at` is a real instant (the canvas strip round-trips local↔UTC), while `due_date` is a calendar day stored as UTC midnight (the strip's date input slices the ISO date).
**Decision:** A `LEDGR_TIMEZONE` env var (IANA name, default `America/New_York`) defines "today"; `src/lib/today.ts` computes the day's bounds with hand-rolled `Intl.formatToParts` math (a guess-and-correct pass that survives DST, no date library). Meetings filter on the timezone's real midnights; due-date comparisons use plain UTC midnights matching the storage encoding. `getTodayData` batches the screen's three queries (meetings, due/overdue open tasks, recent) in one `Promise.all`, all body-free `listColumns`.
**Why / alternatives:** Deriving the timezone from the browser was rejected: day-scoped *queries* run on the server, and a cron (morning agenda, Phase 2) has no browser. Normalizing both fields to one encoding was rejected as a rewrite of working slice-8 semantics; the two-boundary rule is small and contained in one file. A date library (Luxon, date-fns-tz) was rejected under rule 5 for what is ~30 lines of `Intl`. The env var becomes a per-user column if Ledgr ever goes multi-user.
**Affects:** `src/lib/today.ts`, `src/app/page.tsx` (Today replaces the interim home; the type-grouped list moved to `/items`), `src/components/today/QuickCapture.tsx`, `.env.example`, runbook §1.

## ADR-009: Navigation shell as a hardcoded slot table; desktop bar-vs-sidebar resolved by a live toggle
**Date:** 2026-06-12
**Status:** accepted
**Context:** PRD §4.12 wants user-chosen nav slots (Home locked to slot 1, badge counts) with mobile settled on a floating bottom bar and desktop deliberately undecided (open Q9: same bar vs right sidebar), to be settled by trying both.
**Decision:** `src/lib/nav.ts` is a hardcoded slot table (Home, Inbox+badge, Items), the same Build-surface seam pattern as `canvas-fields.ts`; a server `Nav` wrapper resolves the owner (no nav signed out) and fills badge counts, and a client `NavShell` renders the chrome. Both desktop candidates are built behind the same slot model and a toggle button *in the nav itself* flips between them, persisting to localStorage via `useSyncExternalStore` (server renders the bar default; the stored preference applies after hydration). Mobile always gets the bottom bar. Badges re-render on every `router.refresh()`, which all mutation components already call.
**Why / alternatives:** A DB-backed preference was rejected for a single-user A/B trial: localStorage costs nothing, needs no schema, and the loser gets deleted along with the toggle once Q9 is decided (the keeper may then become a Build-surface setting). Building only one candidate was rejected because the PRD explicitly calls for a try-both decision. Icons are three hand-rolled SVGs, not an icon library (rule 5).
**Affects:** `src/lib/nav.ts`, `src/components/nav/` (Nav, NavShell), `src/app/layout.tsx`.

## ADR-010: Inbox membership is an explicit boolean column, set on arrival, cleared only by deliberate triage
**Date:** 2026-06-12
**Status:** accepted
**Context:** The Inbox (PRD §4.2) holds "items that arrived untriaged, awaiting type/entity/date assignment," but Phase 1's only arrival path is quick capture and the PRD doesn't define what makes an item leave.
**Decision:** A real `items.inbox` boolean (not null, default false) with a partial index (`owner_id where inbox and deleted_at is null`). Arrival paths set it (quick capture today; email-in, Todoist pull-ins, and the share target reuse it in Phase 2). It clears only when a control says so: the Inbox row's "✓ Triaged" button or any explicit `inbox: false` PATCH. The nav badge is `countInbox`, riding the partial index.
**Why / alternatives:** A derived definition ("task with no due date or entity") was rejected: a deliberately dateless someday-task would squat in the Inbox forever, and assigning a date would silently "triage" items behind Brandon's back, violating deterministic-and-predictable behavior. A `properties` JSONB key was rejected because the flag is hot (the badge counts it on every page render) and hot fields are columns (ADR-003). Auto-clearing on any edit was rejected for the same predictability reason; if real use shows the explicit click is friction, auto-clear-on-date/entity-assignment is a one-line loosening, and the reverse migration would be impossible to get right.
**Affects:** `src/db/schema.ts`, `drizzle/0001_*.sql`, `src/lib/items.ts` (input/patch/list/`countInbox`), `src/lib/api.ts`, `src/app/api/items/route.ts`, `src/app/inbox/`, `src/components/inbox/TriageControls.tsx`, `src/components/today/QuickCapture.tsx`, `src/lib/nav.ts`, schema.md.

## ADR-011: Per-type lists ride a ViewFilter shape a `views` row can store; the URL carries the filter
**Date:** 2026-06-12
**Status:** accepted
**Context:** Slice 12 needed per-type list pages now, but PRD §4.2 makes every view a stored View Definition in Phase 2; building pages against ad-hoc query params would mean a rewrite when the view builder lands.
**Decision:** `src/lib/views.ts` defines `ViewFilter` (type, status, urgency, kind, due window, entityId) and `ViewSort`, exactly the shapes a `views` row's `filter`/`sort` jsonb will store, plus one query builder over the body-free `listColumns`. Pages (`/tasks`, `/meetings`, `/notes`, `/links`, `/entities`) parse URL search params into that shape and `FilterBar` selects write params back, so the URL *is* the filter state (shareable, back-button-friendly, zero client list state). The entity filter is an EXISTS over `relations` in both directions, **confirmed edges only** (suggested edges stay out of trusted queries, PRD §3.3). Due windows reuse ADR-008's UTC-midnight calendar encoding; date sorts push nulls last (undated tasks sink). `/items` stays as the all-type sweep and the Trash's home; a tab strip links the six pages; Tasks joins the nav slots.
**Why / alternatives:** Client-side filter state was rejected (server-rendered + URL is simpler and matches every existing page). Filtering through suggested edges was rejected: a provisional calendar match must not silently shape a task list. Creating real seeded `views` rows now was rejected as premature; the shape is the contract, the storage rides in with the view builder.
**Affects:** `src/lib/views.ts`, `src/app/{tasks,meetings,notes,links,entities}/`, `src/components/lists/` (ListPage, ListTabs, FilterBar), `src/app/items/page.tsx`, `src/lib/nav.ts`, `src/components/nav/NavShell.tsx`.

## ADR-012: Search binds raw input through `websearch_to_tsquery`; snippets via `ts_headline` over capped `body_text`; the date filter is `updated_at`
**Date:** 2026-06-12
**Status:** accepted
**Context:** Slice 13. PRD §4.2 wants full-text search filtered by type/entity/date but pins neither query syntax, nor snippets, nor which date "date" means.
**Decision:** `GET /api/search` binds the user's raw string through `websearch_to_tsquery('english', …)` (Google-ish syntax: words, quoted phrases, OR, -exclusions; never throws on any input), matches the stored generated tsvector (ADR-003) on `items_search_gin`, ranks with `ts_rank`. Filters: type, entity (confirmed edges, both directions, same fragment as ADR-011), and an `updated_at` day-window in `LEDGR_TIMEZONE`. Results are body-free list rows plus a snippet `ts_headline` computes **in the database** over `left(body_text, 4000)` with `[[..]]` markers the client renders as `<mark>`; markerless headlines (title-only hits) are dropped as noise. The client debounces 300ms and aborts stale requests.
**Why / alternatives:** The snippet is a deliberate, bounded brush with body content on a list-shaped read: the no-body rule's point is never shipping whole bodies to lists, and an ~18-word db-computed excerpt honors that while making results scannable; returning no snippet was rejected as a worse product for zero real savings. `plainto_tsquery` was rejected (no phrases or negation). Filtering on due/meeting dates was rejected for v1: "that note from March" means recency of touch, and the per-type lists already filter their own date fields.
**Affects:** `src/lib/search.ts`, `src/app/api/search/route.ts`, `src/app/search/page.tsx`, `src/components/search/SearchClient.tsx`, `src/lib/nav.ts`.

## ADR-013: Quick capture is a nav-mounted modal with `q` / `Ctrl+K` shortcuts; captures always arrive `inbox: true`
**Date:** 2026-06-12
**Status:** accepted
**Context:** Slice 14 (PRD §4.4): a global "new item" affordance on every screen with a desktop shortcut, title-only creation, date/urgency/entity optional inline.
**Decision:** `NavShell` (already on every signed-in page) owns a "New" button and the global keydown: `q` opens the capture modal (inert while typing — inputs, selects, contentEditable), `Ctrl/Cmd+K` navigates to `/search`. The modal: title (Enter submits and closes), type select defaulting to task, optional due date and urgency. It always posts `inbox: true`, even with fields set: per ADR-010, leaving the Inbox is a deliberate act, never a side effect of capture-time detail. Its Esc handler claims the key in the **capture phase** (preventDefault + stopPropagation) so a capture opened above the item-canvas modal (which closes on any unclaimed Esc at document level, ADR-007) closes alone, one layer per press. Entity assignment at capture is deferred until a relations write path exists (the backlinks slice builds it).
**Why / alternatives:** `q` follows Todoist's quick-add key (Brandon's existing muscle memory; browser `Ctrl+N` can't be intercepted), `Ctrl+K` is the Notion-default for search. ## ADR-015: Relations write path — un-relate hits the pair in both directions; mention edges belong to the body; relating over a suggested edge confirms it
**Date:** 2026-06-12
**Status:** accepted
**Context:** Slice 15 (PRD §4.9): the backlinks panel needed the first relations write path (relate, un-relate, confirm/reject for suggested edges), and the deferred entity-at-capture rides it. The `relations` table is direction-ful (source → target) and role-ful, but the panel the user acts on is direction-blind and dedupes a pair's edges into one row.
**Decision:** `POST/PATCH/DELETE /api/items/[id]/relations` over `relateItems`/`confirmRelations`/`unrelateItems` in `src/lib/relations.ts`. (1) **Un-relate matches the gesture, not the storage:** it deletes every edge between the pair in both directions ("remove = un-relate, never delete", PRD §4.9). (2) **Mention edges (role `mention`) are body-owned and untouchable here:** the write path refuses to create or delete them, because mention diff-sync (slice 5) re-derives them on every body save — a manually deleted mention edge would silently resurrect. Mention-only rows render an `@` marker and no remove control; removing the @-mention in the body is the remove gesture. (3) **Relate upserts on the (source, target, role) unique and flips suggested → confirmed:** relating *is* the confirm gesture and POST is idempotent. Confirm (PATCH) flips every suggested edge between the pair; reject (DELETE `?suggested=true`) deletes only suggested ones, leaving confirmed edges alone. Manual relate writes source = the item the user is on → target = the picked item, role `related` (PRD §3.4 tag direction). Quick capture POSTs the item, then best-effort relates it to the picked entity: a failed relate never blocks the capture, which already landed in the Inbox where triage catches the missing link.
**Why / alternatives:** Deleting only the directed edge was rejected: the panel collapses both directions into one row, so a ✕ that removed half the pair would read as a no-op. Letting un-relate take mention edges was rejected for the resurrect problem. Addressing edges by relation id from the client was rejected: the UI never surfaces rows, it surfaces pairs.
**Affects:** `src/lib/relations.ts`, `src/app/api/items/[id]/relations/route.ts`, `src/components/relations/` (RelatedPanel, RelationActions, AddRelation — replacing `src/components/entity/RelatedItems.tsx`), `src/components/canvas/ItemCanvas.tsx`, `src/components/capture/CaptureModal.tsx`.

**Context:** Slice 22 (PRD §5.1): poll the next 14 days, auto-create meeting items, dedupe on `ms_event_id`, handle reschedule and cancel. The engine never sees Graph JSON; the source normalizes to a `CalendarEvent`. (5) **Reschedule updates `meeting_at`; cancel flags `properties.calendar.canceled = true` and never deletes** (prep survives, PRD §5.1). Cancellation lives in `properties`, not a column: it isn't a hot filter in v1, so keeping it off-schema avoids a migration (hot-fields-are-columns, ADR-003). An event cancelled before we ever synced it creates no tombstone. (6) **Write only on real change.** `unchangedSince` compares title, `meeting_at`, the canceled flag, and Graph's `lastModified`; an unchanged poll writes nothing — which matters because any write bumps `updated_at` and would re-export the item (ADR-017). (7) **`properties.calendar` stores attendees structured** (`{name,email}[]` + a flat `attendeeEmails`) plus organizer/location/joinUrl/webLink/seriesMasterId/bodyPreview, so (a) matchers read attendee emails directly and (b) those string values fall into the FTS document (`jsonb_to_tsvector`, ADR-014) — "Roger" now finds his meetings. Writes **merge** into `properties` so a matcher-written key (matched template, default urgency) survives a later sync. (8) **Two routes, same engine:** `GET /api/machine/calendar-sync` (cron-scoped token, called 6h by `.github/workflows/calendar-sync.yml`) and the user-authed `POST /api/calendar/sync` ("sync now"). A Graph **403** (Calendars.Read / Application Access Policy not yet in place, §1c) is returned as a **503 "not configured"** with a warn log, *not* a captured error, so `error_log` isn't spammed every 6h before setup; `/health` `lastCalendarSyncAt` staying null is the canary. (9) **Soft-deleted meetings are never resurrected or duplicated:** the dedup lookup includes trashed items, and the engine skips writes to them.
**Why / alternatives:** `/events` rejected (returns series masters, not instances). Delta sync rejected for v1 (moving-window complexity for no real gain at this volume). Inbox-on-arrival for calendar rejected (recurring-meeting noise; see (3)). A `canceled` column rejected (migration for a non-hot field). Routing the create through `createItem` rejected: it can't set `ms_event_id` (a machine column), so the engine does owner-scoped raw writes, exactly as the export engine sets `exported_at` — machine columns are written by their owning subsystem.
**Verification:** 24/24 in `scripts/verify-calendar-sync.mts` against Neon with a stub source under a throwaway owner — create, idempotence (no `updated_at` churn), reschedule, cancel/uncancel (item survives, flag flips), dedupe, trashed-item protection, properties-merge (matcher key survives), **attendee FTS** (a unique attendee name is found via `websearch_to_tsquery` over the generated search column), owner scoping, job_state. Plus a **live** `GraphCalendarSource.listEvents(14)` against real Graph returning a clean `GraphError(kind:request, status:403)` — proving the URL/params/`Prefer` header/pagination/error handling are correct; it will return events once §1c is done. `tsc --noEmit` clean.
**Affects:** `src/lib/calendar/` (types, graph-source, sync, owner — all new), `src/app/api/machine/calendar-sync/route.ts`, `src/app/api/calendar/sync/route.ts`, `src/app/health/route.ts` (calendar canary), `.github/workflows/calendar-sync.yml`, runbook §§1c, 2, 2a, `scripts/verify-calendar-sync.mts`.

## ADR-024: Matchers — fixed kind precedence with fuzzy as last resort; engine-made edges suggested by default; matchers run on meeting create only
**Date:** 2026-06-13
**Status:** accepted
**Context:** Slice 23 (PRD §5.1): the deterministic rules that map a calendar event to entities/templates. The `matchers` table was deliberately deferred in Phase 1 (ADR-003); this slice creates it and the engine. PRD pins the condition order (attendee-email → series-id → title-regex → fuzzy, fuzzy only when nothing better matched) and that attendee/fuzzy matches land `suggested`, but leaves: how series-id/regex trust is set, whether multiple rules accumulate, when matchers re-run, and how a suggested auto-match interacts with an already-confirmed edge.
**Decision:** (1) **`matchers` table + `pg_trgm`** via migration 0004 (extension added by hand — drizzle-kit doesn't emit `CREATE EXTENSION`, same as ADR-014's GIN index). `condition` and `action` are validated jsonb; a bad regex is rejected at **save time** (`validateCondition`), never left to throw mid-sync. (2) **Fixed kind precedence, fuzzy gated.** `CONDITION_RANK` (attendee 0, series 1, regex 2, fuzzy 3) plus the user `priority` orders matched rules; **fuzzy candidates are evaluated only when no attendee/series/regex rule matched** (PRD's "last resort, external guests / room-only invites"). Fuzzy uses Postgres `similarity()` (default threshold 0.3), no new dependency. (3) **All matched rules accumulate** (a meeting can pick up several entities/tags), and for the same entity **confirmed wins over suggested**; template/urgency take the highest-precedence rule that sets them. (4) **Trust by condition kind:** attendee-email and fuzzy → `suggested` (PRD), series-id and title-regex → `confirmed` (a specific series is unambiguous; a regex is user-authored). A rule's `action.matchState` overrides. (5) **`addMatchEdge` never downgrades:** a suggested auto-match over an existing confirmed (or manually-related) edge leaves it confirmed (`ON CONFLICT … case when excluded.match_state='confirmed' or relations.match_state='confirmed' then confirmed else suggested`); a confirmed auto-match upgrades a suggested edge. (6) **Matchers run on meeting CREATE only**, wired through a `runCalendarSync` `onCreated` callback the routes supply (the sync engine stays matcher-agnostic and verifies independently). Not on every update — re-running would resurrect a suggestion the user rejected (the same hazard mention-sync avoids). The matched template + matcher ids are recorded in `properties.match` for the prep-template slice (24) and learn-by-confirmation. (7) **Edges are `meeting → entity`, role `related`** (same key as a manual relate, so they merge and the backlinks panel collapses the pair). urgency is carried in the result but **not written to meetings** (no UI surface, ADR-018). (8) A minimal user-authed API (`GET/POST /api/matchers`, `DELETE /api/matchers/[id]`) so rules are manageable; the full Build-surface UI is Phase 3.
**Scope / deferred:** the **engine, config, CRUD, and suggested/confirmed states are complete and verified**. The **setup wizard** (sample recent/upcoming events → pick matches → write first rules) is deferred — it needs live calendar reads, which are §1c-blocked. **Learn-by-confirmation** (confirming a coincident match offers to save a standing rule) is deferred — it's a UI affordance on the slice-15 confirm path; the data backbone (matched matcher ids in `properties.match`, the matchers API) is in place. Both are tracked as a follow-up; the roadmap line stays `[~]`.
**Why / alternatives:** Auto-resolving attendee emails to entities by a stored entity email was rejected for the PRD's explicit condition→action rule model (the rule names the entities). Re-running matchers on every sync was rejected (resurrects rejected suggestions). Making series/regex `suggested` too was considered but rejected — they're deterministic/user-authored, and forcing a confirm on every instance of a series Brandon already set a rule for is friction; `action.matchState` covers the cautious case.
**Verification:** 16/16 in `scripts/verify-matchers.mts` against Neon with **real pg_trgm** under a throwaway owner — kind precedence, fuzzy-as-last-resort gating (a regex hit suppresses a would-match fuzzy rule), confirmed-wins accumulation, save-time regex rejection, edge writing with the right `match_state`, no-downgrade of a confirmed edge, `properties.match` recording, delete + owner scoping. `tsc --noEmit` clean; calendar verification still 24/24 after the `onCreated` seam.
**Affects:** `src/db/schema.ts` (+`matchers`), `drizzle/0004_*` (+pg_trgm), `src/lib/matchers/` (types, store, engine — new), `src/lib/relations.ts` (`addMatchEdge`), `src/lib/calendar/sync.ts` (`onCreated`), both calendar routes, `src/app/api/matchers/` (new), schema.md, `scripts/verify-matchers.mts`.

## ADR-026: Todoist sync — batched engine behind a client interface; three-way due reconcile; webhook triggers an idempotent full sync
**Date:** 2026-06-13
**Status:** accepted
**Context:** Slice 25 (PRD §5.2): push dated tasks to Todoist, sync completions + date changes back, pull in Todoist-inbox tasks (offline capture), delegate recurrence to Todoist, Ledgr canonical for content. The PRD pins behavior but not: where the push fires (on save vs. on sync), how the conflict rule is implemented concretely, or how the webhook relates to polling.
**Decision:** (1) **A `TodoistClient` interface** (REST v2 HTTP client in prod; a stub verifies the engine with no token) — same provider discipline as `CalendarSource`/`ExportTarget`. (2) **Batched sync, not inline-on-save.** The push/pull engine runs on the cron (polling fallback), the webhook, and "sync now" — never inline in `createItem`/`updateItem`. Saves stay fast and offline-safe; an external call on every keystroke-save would be fragile. The webhook makes the push prompt enough for notifications; tightening the poll or adding an inline fire-and-forget push is a noted refinement. (3) **Push = dated open tasks only** (content, due, priority, a link back in the description); undated tasks stay app-only, and a task that **loses its due is deleted from Todoist and unlinked** (honors "undated stays app-only"). Ledgr urgency maps to Todoist priority (critical→4 … low→1, none→1). (4) **Conflict rule via a three-way due reconcile.** `properties.todoist = {id, syncedDue}` records the due at last sync; comparing Ledgr / Todoist / syncedDue lets **Ledgr win when both changed** (the PRD rule) while still accepting a **Todoist-only date change** back into Ledgr. Content is Ledgr-canonical — always pushed, never pulled. (5) **Completion both ways:** a Ledgr task marked done closes its Todoist task; a linked task **gone from Todoist's active list** marks the Ledgr task done. Disappearance = completion (deletion is the same signal; accepted for v1). **Recurrence is Todoist's entirely** — a recurring task stays active after each completion, so per-occurrence completion logging would need the Sync API's completed-items endpoint; **deferred** (noted). (6) **Inbox pull-in:** active Todoist tasks in the Inbox project with no Ledgr link are imported as `inbox: true` tasks (the offline-capture path, ADR-010), and never re-imported (the link excludes them). (7) **The webhook** (`POST /api/todoist/webhook`, public from Clerk, HMAC-SHA256 verified with the app client secret) triggers **one idempotent `runTodoistSync`** rather than duplicating per-event logic — the polling engine is the single source of truth, the webhook just makes it prompt. It returns **200 even on a sync error** (Todoist retries non-2xx; the cron is the backstop, so a retry storm is worse than a missed beat). (8) `/health` gains `lastTodoistSyncAt`/`lastTodoistRunAt` (the export/calendar canary pattern).
**Why / alternatives:** Inline push-on-save rejected (fragile, slow, offline-hostile). Per-event webhook handlers rejected (logic duplicated across webhook + poll; the idempotent full sync is simpler and self-healing). Treating disappearance as completion rejecting deletion was considered but deletion-vs-completion can't be told apart from the active list alone without the Sync API; accepted for v1. A `todoist_completed` column rejected — completion is just `status`.
**Verification:** 14/14 in `scripts/verify-todoist-sync.mts` against Neon with a stub client under a throwaway owner — push-create (dated only, priority mapped), undated-skip, idempotence (zero ops on a no-change re-run), Ledgr-canonical content push, the three-way due reconcile (Todoist-only change syncs back; both-changed → Ledgr wins), completion both ways, lost-its-due delete+unlink, inbox pull-in + no-re-import, job_state. `tsc --noEmit` clean. The REST client + webhook HMAC verification are built but exercised only via the stub; the live path needs the **Brandon-steps** (token, app client secret, webhook registration, `LEDGR_CRON_TOKEN`).
**Affects:** `src/lib/todoist/` (types, client, sync, owner — new), `src/app/api/machine/todoist-sync/route.ts`, `src/app/api/todoist/sync/route.ts`, `src/app/api/todoist/webhook/route.ts`, `src/proxy.ts` (webhook public), `src/app/health/route.ts` (canary), `.github/workflows/todoist-sync.yml`, `.env.example` (`TODOIST_CLIENT_SECRET`, `TODOIST_OWNER_UPN`), runbook §§1d, 2, 2a, 3, `scripts/verify-todoist-sync.mts`.

## ADR-027: Email-in — messages/delta over a MailSource; HTML reduced to text-paragraph BlockNote; move-out + properties dedup; storage gains a server-side put
**Date:** 2026-06-13
**Status:** accepted
**Context:** Slice 26 (PRD §5.3): poll the "Ledgr Import" Outlook folder via Graph `messages/delta`, turn each message into a `note` (or `task` if subject is `task:`-prefixed), convert the body, store attachments to R2, mark-read + move so nothing double-imports. A `email_message_id` column for dedup rejected — `properties.email.messageId` rides the existing GIN index and avoids a migration; move-out is the real guard anyway. Proxying attachment bytes through a presigned browser-style flow makes no sense with no browser; the server-side `putObject` is the right primitive (and it's behind the same provider interface, so the local build still swaps cleanly).
**Verification:** 20/20 in `scripts/verify-email-in.mts` — a **real R2 `putObject`/`deleteObject` roundtrip** against the live bucket, then the engine under a stub source: note/task routing, `task:` strip, HTML→text body, `inbox:true`, sender in `properties.email`, dedup (re-seen message skipped, single item), graceful attachment-drop without storage, delta-token persistence and start-from-token, and **an errored run not advancing the token** (with the item surviving for next-run dedup). `tsc --noEmit` clean. ## ADR-034: Push notifications — Web Push hand-rolled over node:crypto behind a PushSender; daily agenda on Vercel cron, hourly prep on GitHub Actions
**Date:** 2026-06-13
**Status:** accepted
**Context:** Slice 30 (PRD §4.5/§4.11): web push for the morning agenda summary and meeting-prep-ready notices (task *reminders* stay Todoist's job). Web Push needs a VAPID keypair, a subscriptions store, a service-worker push handler, and message encryption (RFC 8291). The obvious move is the `web-push` npm package; rule 5 says justify it.
**Decision:** (1) **Hand-rolled over `node:crypto`, no dependency.** The whole protocol is in `node:crypto`: VAPID is an ES256 JWT (`src/lib/push/vapid.ts`), and message encryption is ECDH-P256 + HKDF-SHA256 + AES-128-GCM per RFC 8291/8188 (`src/lib/push/encrypt.ts`). This matches every prior boring-crypto call (hand-rolled HMAC for the Todoist webhook, the PNG icon encoder, the markdown/print serializers, `aws4fetch` over the AWS SDK) and keeps the dep count flat. The risk — fiddly crypto that fails silently — is bought down by a **round-trip self-test** in `verify-push.mts`: encrypt to a freshly generated subscription, decrypt from its side, assert the plaintext matches; plus a VAPID JWT signed then verified with the public key. (2) **A `PushSender` interface** (`WebPushSender` in prod; a stub in verification) — same provider discipline as `CalendarSource`/`MailSource`/`TodoistClient`/`ExportTarget`. `getWebPushSender()` returns null when VAPID keys are unset (the "not configured" path, like Graph/Todoist). (3) **VAPID keys live in env** (`VAPID_PUBLIC_KEY`/`VAPID_PRIVATE_KEY`/`VAPID_SUBJECT`), generated once by `scripts/make-vapid-keys.mjs`; the public key is also the browser's `applicationServerKey`, served at runtime by `GET /api/push` (no `NEXT_PUBLIC_` rebuild needed). (4) **`push_subscriptions` table** (migration 0006), owner-scoped, unique on `endpoint`; subscribe is an idempotent upsert, and a push-service **Gone (404/410) prunes the row** so the table self-heals. (5) **Two triggers, one send path.** The morning agenda is a deterministic count summary (no model — rule 3; the AI-assembled agenda is Phase 3) built from `getTodayData`, sent once/day with a day-guard, on the **Vercel daily cron** (`/api/machine/notify-agenda`, 11:00 UTC ≈ 7am ET — Hobby supports daily). Meeting-prep-ready notifies each meeting coming due within a 2h window **that has a confirmed related entity** (so there's prep worth opening), once per meeting via a `properties.notify.prepNotifiedAt` stamp, on an **hourly GitHub Actions** poll (`/api/machine/notify-prep`) — sub-daily, so it uses the same cron-token door as calendar/email. (6) **SW v3** gains `push` (renders `{title,body,url,tag}`) and `notificationclick` (focus-or-open the url) handlers; it still never touches any cache. (7) `/health` gains `lastAgendaNotifyAt`/`lastPrepNotifyAt` canaries. (8) A `PushToggle` on Today enables/disables for the current browser; renders nothing when push is unsupported/unconfigured (delivery needs the prod-registered SW).
**Why / alternatives:** `web-push` rejected on rule 5 (a dependency + transitive tree to do what `node:crypto` already does, verifiable by round-trip). Inline send-on-save rejected (the existing sync pattern is batched/cron-driven; notifications aren't latency-critical). Notifying every upcoming meeting rejected as noisy — the confirmed-entity gate is the "prep is meaningful" signal. Storing notified meeting ids in `job_state` rejected for the per-item flag (bounded, no growth). Putting the daily agenda on GitHub Actions rejected — Vercel cron does daily natively; only the sub-daily prep poll needs Actions.
**Verification:** 25/25 in `scripts/verify-push.mts` against Neon under a throwaway owner with a stub sender — VAPID keygen shape, JWT sign+verify (and tamper-fails), the **RFC 8291 encryption round-trip**, store CRUD (upsert, unsubscribe), send-and-prune of a Gone endpoint, the agenda day-guard (second same-day run skips), prep gating (in-window+entity notified; no-entity / out-of-window / canceled skipped; re-run doesn't re-notify), and owner scoping. `tsc --noEmit` clean; full `next build` clean (new routes compile). Live end-to-end (a real browser subscription + a delivered notification) is a Brandon-step once VAPID keys are set (runbook §1e).
**Affects:** `src/db/schema.ts` (+`push_subscriptions`), `drizzle/0006_*`, `src/lib/push/` (types, vapid, encrypt, web-push, store, notify, owner — new), `src/app/api/push/route.ts` + `subscribe/route.ts`, `src/app/api/machine/notify-agenda/` + `notify-prep/`, `src/components/pwa/PushToggle.tsx`, `src/app/page.tsx`, `public/sw.js` (v3), `src/app/health/route.ts`, `vercel.json`, `.github/workflows/notify-prep.yml`, `scripts/make-vapid-keys.mjs`, `.env.example`, runbook §§1e, 2, 2a, `scripts/verify-push.mts`.

## ADR-035: Public share links — unguessable token → the shared print render; revocation is a stamp; the print document shell is now shared
**Date:** 2026-06-13
**Status:** accepted
**Context:** Slice 31 (PRD §4.12, a named Notion pain point): read-only, print-friendly, PDF-downloadable public links. The PRD says they're "statically rendered/cached" (§6.5). Open: what the public path renders, how revocation works against caching, and avoiding a second renderer.
**Decision:** (1) **One render, reused.** The self-contained document shell (inline CSS, dark on screen / black-on-white in print, the Print/PDF button) moved from the Save Offline print route into `renderPrintDocument()` in `print-html.ts`; both the owner's `/items/[id]/print` and the public `/share/[token]` call it, so a pinned offline copy and a shared link are byte-identical renders. The share page adds a small "Shared from Ledgr · read-only" footer (dropped under `@media print`). Mentions already render as plain names (no `ledgr://` nav), so a shared page leaks no path into the owner's data. (2) **`share_tokens` table** (migration 0007): an unguessable 24-byte base64url `token` (~192 bits, the same security posture as the machine tokens) maps to one item. Issuance is owner-scoped and mints a fresh token each call; **revocation is a `revoked_at` stamp, not a delete** (auditable, and a revoked string can't be silently reissued). Cascade-deletes with the item at purge. (3) **The public path takes no Clerk** (`/share(.*)` joins the public-route set in `proxy.ts`, alongside the Todoist webhook); the token is the credential. `resolveShareToken` joins token→item in one query so it only yields a live token on a live item — a revoked token or trashed item is a plain 404. **Issuance/listing/revocation (`/api/items/[id]/share`) stays Clerk-protected.** (4) **Caching vs revocation:** the render sets `public, s-maxage=60, stale-while-revalidate=300` — cacheable at the CDN (PRD §6.5, cheap origin) but revocation propagates within ~60s at the edge and is immediate at the origin. Plus `X-Robots-Tag: noindex` and `Referrer-Policy: no-referrer` so a shared link doesn't leak into search or referrers. (5) A `ShareLink` control on every canvas (beside Save Offline) creates/copies/revokes links; the client builds the absolute URL from the returned token (env-agnostic).
**Why / alternatives:** A separate share renderer rejected (forking the print render; extracting the shell unifies them). Revoke-as-delete rejected (loses history; the stamp also blocks reissue races). Long CDN cache rejected (revocation is a security property; 60s is the balance the PRD's "cached" intent allows). Embedding owner-scoped checks on the public path rejected — there's no session; the token + live-item join is the boundary. Storing the share URL base in env rejected (the client knows its own origin).
**Verification:** 18/18 in `scripts/verify-share.mts` against Neon under a throwaway owner — issuance, public resolve (token→item+body), the shared render (full HTML doc, body HTML-escaped, the read-only footer), listing, independent tokens, revocation (then resolves null) + idempotent re-revoke, trashed-item protection, unknown token → null, and owner scoping (a non-owner can't share/list/revoke, and a foreign revoke attempt leaves the link live). `verify-print.mts` still 23/23 after the shell extraction; `tsc` + full `next build` clean.
**Affects:** `src/lib/print-html.ts` (`renderPrintDocument` + shared CSS), `src/app/items/[id]/print/route.ts` (refactored onto it), `src/db/schema.ts` (+`share_tokens`), `drizzle/0007_*`, `src/lib/share.ts` (new), `src/app/share/[token]/route.ts` (new, public), `src/app/api/items/[id]/share/route.ts` (new), `src/proxy.ts` (public route), `src/components/canvas/ShareLink.tsx` (new) + `ItemCanvas.tsx`, `scripts/verify-share.mts`.

## ADR-037: The Markdown epoch — Markdown-canonical body, bespoke-first types, per-type canvas, two-builder model
**Date:** 2026-06-13
**Status:** accepted — **the turning point. Supersedes the body-format and "Notion-default" decisions of PRD §10 and revises ADR-006 (serializer role) and ADR-007 (single canvas).** Marked by git tag `v0.17-blocknote-canonical` (last pre-pivot commit), PRD bumped to v0.18 "Markdown epoch," and a GitHub Release.

## ADR-038: Markdown-native editor — adopt Tiptap with the first-party `@tiptap/markdown` extension
**Date:** 2026-06-13
**Status:** **accepted** (2026-06-13) — CORE decision, **both-agree satisfied: Brandon recommended, Tyler agreed** (CLAUDE.md "Building together"). M2 (stand-up + round-trip proof) is now unblocked.
**Context:** Roadmap Phase M, slice **M1 — editor selection**. ADR-037 made Markdown the canonical body format and left the editor library TBD (Tiptap / Milkdown / Lexical), with a stated lean toward Tiptap. M1's job is to pick one against concrete criteria and record why; the actual stand-up + round-trip proof is M2. The editor must preserve what the v0.17 BlockNote path already does and the markdown serializer (`src/lib/markdown.ts`) already emits:
- **Lossless markdown round-trip** (type → markdown → reload → identical), because markdown is now the source of truth, not a derived export.
- **Color/highlight marks** that serialize to the pinned inline-HTML encoding in `src/lib/colors.ts` — `<span style="color:…">` for text color and `<mark class="hl-<name>" style="background-color:…">` for highlight (9 Notion-palette colors). This inline-HTML-in-markdown choice is *why* sermon colors survived the pivot (ADR-037); the editor must reproduce it exactly.
- **`@`-mention** as the markdown link the serializer already produces: `[@Title](ledgr://item/<uuid>)`. The mention carries `{itemId, title}`; `src/lib/mentions.ts` diffs these out of the saved body into `relations` rows (role `mention`) on every save, so the editor's mention node must expose a stable item id.
- **Lazy-load + bundle discipline** (Principle 8 / PRD §6.4): the editor already loads only through `LazyEditor` (code-split, client-only) and must keep doing so.
- **A credible path to the later "Notion feel"** (slash menu, block drag — wanted, explicitly *not* a v1 requirement per ADR-037) and to Obsidian-style `^block-id` anchors (a Phase-3 feature, `explorations/block-linked-action-items.md`; M1 only keeps the door open).
**Decision:** Adopt **Tiptap** (ProseMirror under a maintained React API) with its **first-party `@tiptap/markdown` extension** as the markdown-native canvas, replacing BlockNote when M2–M4 land. Rationale on each criterion:
1. **Round-trip:** `@tiptap/markdown` is *bidirectional* and round-trip-tested (parse markdown → Tiptap JSON → serialize back), driven via `setContent(md, {contentType:'markdown'})` / `editor.getMarkdown()`. It is now first-party (not the older community `tiptap-markdown`), which is the Principle-5 maintenance answer.
2. **Color marks:** custom marks define a `renderMarkdown(node, helpers)` hook that returns an arbitrary string, so a `textColor`/`highlight` mark emits our exact `colors.ts` inline-HTML verbatim, with a matching `parseMarkdown` handler for the reverse. **This is the one genuinely bespoke piece** — round-trip of custom marks needs the matching parse handler or it's lossy. Bounded and known; proven in M2.
3. **Mentions:** Tiptap ships an official `Mention` extension (the `@`-suggestion UX is off-the-shelf); we give it an `itemId` attribute and a `renderMarkdown` that emits `[@Title](ledgr://item/<id>)`, so `mentions.ts` keeps working unchanged.
4. **Bundle:** headless + tree-shakeable extensions, stays behind `LazyEditor`.
5. **Notion-feel + block anchors:** Tiptap has *official* slash-command (suggestion), drag-handle, and node-id extensions — the clearest off-the-shelf path to the deferred polish, and node ids are the natural seat for `^block-id` later.
6. **Team fit (decisive):** Tiptap is **Savor-proven in Tyler's production app** for the exact Tiptap→markdown pattern, so the largest unknown (custom-mark round-trip) is already walked by a co-builder, and it's the stack he brings to the Papers/Songs modules.
**Why / alternatives:** **Milkdown** — *technically the purest fit*: it's markdown-native by construction (a remark AST *is* its document model), so lossless round-trip is inherent rather than an add-on. Rejected as the pick, not on capability but on ecosystem and fit: bare-bones React integration (hand-built UI for menus/handles), a smaller ecosystem to lean on (Principle 5), and no in-house production proof — against Tiptap's larger ecosystem, off-the-shelf slash/drag/mention extensions, and Tyler's Savor track record. Kept as the documented runner-up: if Tiptap's custom-mark round-trip proves uglier than expected in M2, Milkdown's native-markdown model is the fallback. **Lexical** (Meta) — fastest/lightest and great for huge docs, but markdown is an import/export transform layer, *not* the canonical model, so it's the weakest fit for "markdown is the source of truth," and its lower-level API means more code for the same result with no offsetting team familiarity. Rejected.

## ADR-040: M3 executed — Markdown is the only body path; markdown-it for server render; dev data wiped, not migrated; BlockNote fully removed
**Date:** 2026-06-13
**Status:** accepted. Implements the already-agreed core decisions (ADR-037 body format, ADR-038 Tiptap, ADR-039 direct cutover), so it is execution, not a new core decision. The one genuinely new piece — a server-side markdown-rendering dependency (markdown-it) — is a non-core renderer/Principle-5 call (the roadmap pre-authorized "one vetted markdown lib"); **flagged to Tyler in COLLAB** because it sets the core markdown→HTML path the modules will share.
**Context:** Roadmap **Phase M slice M3** (the merged M3+M4 cutover, ADR-039). M2 stood the Tiptap editor up in isolation; M3 makes markdown the only path: repoint every renderer/reader/writer off BlockNote JSON and onto the `{format,text}` markdown body, then delete BlockNote.
**Decision:**
1. **Body contract → `{format,text}`.** New `src/lib/body.ts` owns the shape: `makeMarkdownBody(text)`, `isItemBody(x)`, and the tolerant `bodyMarkdown(x)` (extracts `.text`, degrades null/foreign to `""`). `items.body` / `revisions.body` stay **jsonb** — no column migration, the shape just changed inside. Every reader/writer goes through `body.ts`.
2. **Server render via markdown-it** (new `src/lib/markdown-render.ts`). One `markdown-it` instance, `html:true` (passes the editor's `<span style>`/`<mark class>` color HTML through), with one token-rewrite core rule that (a) shifts body headings down one level so the document title keeps `<h1>`, and (b) renders `ledgr://item/<id>` links as `<span class="mention">` (a `ledgr://` URI is meaningless on paper/a public link). Exports `markdownToHtml` (print/share) and `markdownToText` (render-then-strip for the FTS `body_text`). **Server-only** — never imported by a client component, so the editor bundle never pays for it. markdown-it is the **one vetted markdown dependency** (Principle 5): single synchronous package, battle-tested; chosen over remark (multi-package, async) for simplicity.
3. **Readers switched:** `print-html.ts` keeps the self-contained document shell + CSS (added `th`) and now renders the body via `markdownToHtml(bodyMarkdown(body))` (the BlockNote JSON walker is gone); `export/engine.ts` emits `bodyMarkdown(item.body)` directly (the body *is* the markdown); `body-text.ts` → `markdownToText(bodyMarkdown(body))`; `mentions.ts` → `collectMentionIdsFromMarkdown` (regex for the `ledgr://item/<id>` href) instead of walking JSON.
4. **Writers switched:** `parseItemPayload` accepts a `{format,text}` object (or null) and rejects the old block array; `ItemEditor` moved to `src/components/markdown-editor/` and wired to `LazyMarkdownEditor` (reads `bodyMarkdown(item.body)`, writes `makeMarkdownBody(markdown)`); email-in emits markdown (`emailToMarkdown`, light inline-escaping).
5. **Data: wiped, not migrated (Brandon's in-session call).** ADR-039 pre-authorized "wipe or convert"; I first converted, Brandon directed **wipe and start fresh** (alpha, nothing to protect). All 30 dev items deleted (cascades cleared relations/revisions/attachments/share_tokens); users + types kept. Because there is no data to migrate, the BlockNote→markdown serializer (`src/lib/markdown.ts`) and the migration script were **deleted, not kept as a tool** — removing the last BlockNote vestige.
6. **BlockNote retired:** `@blocknote/core|mantine|react` uninstalled; `src/components/editor/` deleted; the stale `.bn-root`/`.ledgr-mention` rules dropped from `globals.css` (the editor carries its own CSS).
7. **Security note recorded:** `html:true` is safe under Ledgr's single-trusted-author model (Save Offline is owner-only; a share link publishes the owner's own content; no other user's data exists on an instance). Revisit with a sanitizer only if Ledgr ever renders content authored by someone other than the viewer's owner.
**Side fix (pre-existing, surfaced here):** the OneDrive export engine stamped `exportedAt: new Date()` while the schema's `updatedAt.$onUpdate` set a *second* `new Date()` microseconds later in the same UPDATE, so `needsExportWhere`'s `updatedAt > exportedAt` was true on a millisecond boundary → a spurious re-export on the next run (flaky, ~40% in `verify-export`). Confirmed flaky on pre-M3 code too. Fixed by pinning both stamps to one instant (`set({ exportedAt: now, updatedAt: now, … })`); `verify-export` now 5/5 green.
**Why / alternatives:** *Keep `markdown.ts` as a migration tool* — rejected once Brandon chose wipe: with no data to migrate it is dead BlockNote code, against "start from scratch." *remark instead of markdown-it* — markdown-it is the simpler single synchronous package for a string→string server render. *Sanitize the HTML* — deferred under the single-author model (Principle 5; the editor only ever emits the safe color HTML).
**Verification:** `tsc --noEmit` clean; full `next build` clean (all routes, incl. `/items/[id]`, `/items/[id]/print`, `/share/[token]`). **24/24 verify scripts green** — rewrote `verify-print` (markdown→HTML), added `verify-body-contract` (the `{format,text}` contract + FTS strip + markdown mention scan), and updated the BlockNote-body fixtures in `verify-editor`/`verify-email-in`/`verify-lists-search`/`verify-items`/`verify-relations`/`verify-share`/`verify-export` to markdown. **In-browser (dev stand-in):** the share render shows the heading shift, color span, `<mark>` highlight, mention-as-flat-name (no `ledgr://` leak), code block, and GFM table with no console errors; the real canvas mounts the Tiptap editor over a `{format,text}` body (toolbar, colors, highlight, mention chip), and a live browser edit autosaved back as `{format,text}` markdown with `body_text` reindexed — heading syntax and color HTML intact through the round-trip.
**Affects:** new `src/lib/body.ts`, `src/lib/markdown-render.ts`; rewritten `src/lib/print-html.ts`, `src/lib/body-text.ts`, `src/lib/mentions.ts`, `src/lib/email/html.ts`; edited `src/lib/api.ts`, `src/lib/export/engine.ts`, `src/lib/email/sync.ts`, `src/db/schema.ts` (comments), `src/app/globals.css`, `src/lib/editor/mention-markdown.ts` (+`collectMentionIdsFromMarkdown`); moved `ItemEditor.tsx` into `src/components/markdown-editor/` + `ItemCanvas.tsx` import; **deleted** `src/components/editor/`, `src/lib/markdown.ts`; `package.json` (−`@blocknote/*`, +`markdown-it`/`@types/markdown-it`); verify scripts as above.

## ADR-041: M5 — the per-type canvas seam (a type may declare its own canvas; shell/canvas split, default = markdown)
**Date:** 2026-06-13
**Status:** accepted. **Implements the per-type-canvas pivot ADR-037 already decided with both-agree** ("bespoke-first with per-type canvases"), so the *capability* isn't a new core decision — this ADR records its concrete, minimal shape. Tyler's PR #1 (`ty-additions/FOR-BRANDON-approach-diff.md`, decision #2) names "a system type may declare its own canvas component" as the single most important thing to agree on; ADR-037 said yes. The **registration API** is deliberately *not* finalized here — that's M6, co-designed with the first real module (Papers). Flagged to Tyler in `COLLAB.md` since the seam's shape is shared core.
**Context:** Roadmap **Phase M slice M5**, the platform hook the Songs/Papers modules need. The seam has to let a type render through its *own* canvas while everything without one keeps the markdown canvas unchanged (PRD §4.13 / ADR-037: "types without a canvas get the default markdown canvas"). Scope per the roadmap: ship **default-only plus one trivial proof**, not the full module API.
**Decision:**
1. **Shell / canvas split.** `src/components/canvas/ItemCanvas.tsx` is now a thin **shell**: it owns the universal frame — owner check, item load, trash/`notFound` guards, and the ancestor breadcrumb — then resolves the type's canvas and renders `<Canvas item ownerId variant />`. Everything type-specific moved out.
2. **The default canvas.** `src/components/canvas/MarkdownCanvas.tsx` is the markdown canvas (the markdown editor + the type's standard panels + the backlinks panel + Save Offline + Share + the collapsed Fields section) — a near-verbatim extraction of the old `ItemCanvas` body. Every type without a registered canvas renders through it, so the default experience is unchanged.
3. **Policy split from wiring (for testability).** `src/lib/canvas-registry.ts` is the **pure policy** half — `canvasIdForType(type)` returns a canvas id, defaulting to `"markdown"`; it imports no components (only `import type`), so it resolves identically on the server and in a plain-node verify script. The **wiring** half (canvas id → React component) is a small `Record` in the shell, with a `?? MarkdownCanvas` fallback so a policy/wiring drift can't crash the page. This mirrors how a module will later *declare* `{type → canvas}` (policy) and *provide* the component (wiring) — so the split is M6-shaped, not throwaway.
4. **The one trivial proof: `link`.** `src/components/canvas/LinkCanvas.tsx` registers for the `link` type — it foregrounds the URL as a click-through chip, then composes `<MarkdownCanvas/>` underneath. It exercises the seam (a non-default canvas renders for one type) and demonstrates the exact pattern a real module canvas follows: add a bespoke surface, reuse the standard canvas. Removing the `link` entry from the registry drops links straight back to the default — nothing else depends on it.
5. **Footer-fields extracted + pinned.** The collapsed Fields section's logic moved from `ItemCanvas` into a pure `footerFieldsFor(item)` in `src/lib/canvas-fields.ts` (beside `topStripFields`), so the extraction is one testable place and the ADR-018 "due/urgency are task-only, even on legacy rows" invariant is asserted by a test, not just prose.
**Why / alternatives:** *Thin shell (canvas owns everything visual incl. Related/Share/Fields)* — rejected for M5: maximum module freedom, but every custom canvas would re-implement or forget the cross-cutting chrome. *Thick shell (canvas = only the body editor)* — too restrictive; a chord grid needs to replace more than the editor. The chosen middle — **the canvas is the whole type-specific surface; the shell is load + guards + breadcrumb; shared chrome is composed, not duplicated** — matches Tyler's "canvas = the type's editing UI" and keeps the default DRY (LinkCanvas just wraps the default). *One combined registry returning components* — rejected because importing it would drag the editor bundle into the verify script; the policy/wiring split keeps the policy pure and testable. *Finalize the module-registration API now* — deferred to M6 by the roadmap ("build it to fit the path actually walked," with Papers), so M5 stays a minimal, reversible seam.
**Verification:** `tsc --noEmit` clean; full `next build` clean (all routes compile, incl. `/items/[id]`). New `scripts/verify-canvas-seam.mts` **16/16** — `canvasIdForType` (`link` → its own canvas; `note`/`task`/`meeting`/`entity`/unknown/empty → default) and `footerFieldsFor` (bare note = Type/Created/Updated only; a note with legacy due/urgency still hides them — ADR-018; task/link/entity keep their distinguishing field in the strip, not the footer). No existing verify script imports the touched modules, so no regression surface. **In-browser (dev stand-in, auth-stub):** the `link` item renders the URL chip above the title with the full default canvas beneath it (editor toolbar, URL field strip, Related, Save Offline, Share, Fields), and a `note` renders through the unchanged default canvas (no chip, markdown headings/lists/rules intact) — zero console errors on both.
**Affects:** new `src/lib/canvas-registry.ts`, `src/components/canvas/MarkdownCanvas.tsx`, `src/components/canvas/LinkCanvas.tsx`, `scripts/verify-canvas-seam.mts`; edited `src/lib/canvas-fields.ts` (+`footerFieldsFor`/`FooterField`), `src/components/canvas/ItemCanvas.tsx` (slimmed to the shell + canvas wiring); `.claude/launch.json` (`autoPort: true` on the dev config, so the preview server coexists with a manual dev server). ## ADR-043: M6 — the module-registration boundary (a module contributes `{type, canvas, exporters, integration}`; core is the first module). Phase M complete.
**Date:** 2026-06-13
**Status:** accepted. The **module-system boundary is core** (CLAUDE.md "Building together" → both-agree + ADR). **Brandon directed shipping this solo without waiting on Tyler's sign-off** ("finish off Phase M, push live, no confirmation needed from Tyler on this one"), so the both-agree gate is **explicitly waived by the owner for this merge**; it is designed against Tyler's PR #1 specs and flagged to him in `COLLAB.md` so he can still react. Implements roadmap **M6** and the capability-not-modules scope of ADR-042; the per-type-canvas pivot it builds on was already both-agreed (ADR-037).
**Context:** Phase M's last slice. M5 (ADR-041) split `ItemCanvas` into a shell + the default `MarkdownCanvas` and added the per-type canvas seam as two deliberately-minimal halves: a **pure policy** (`canvasIdForType` in `canvas-registry.ts`, no component imports → node-testable) and a **hardcoded wiring `Record`** (`CANVAS_COMPONENTS` in the shell). M5 explicitly deferred the registration *API* to M6, "built to fit the path actually walked." M6 turns that hardcoded wiring into a real boundary: a *module* (Tyler PR #1 §1 — a workflow packaged as a unit: system type(s) + canonical body format + canvas + exporters + optional integration) contributes its bundle *onto* core rather than editing shared files, with per-user enable as a later config flip.
**Decision:**
1. **`src/lib/modules.ts` — the pure policy/contract half.** Defines the contract: `ModuleManifest = { id, label, enabledByDefault, types: ModuleTypeDef[], exporters: ExporterDef[], integration?: IntegrationDef }`; `ModuleTypeDef = { key, label, canonicalFormat, canvasId, icon? }`; `ExporterDef = { id, label, forType, fileExtension, render }`; `IntegrationDef = { id, label, direction }`. Holds the registry (`registerModule`, `allModules`, the `isModuleEnabled` enable seam) and the resolvers core dispatches through: `canvasIdForType`, `canonicalFormatForType`, `exportersForType`, `moduleForType`, `typeDefFor`, `registeredTypeKeys`. **Kept pure** — imports only `react` types + `@/lib/body` (`MARKDOWN_FORMAT`) + `import type` getItem — so it resolves identically on the server and in a plain-node verify script (the discipline M5 set). The canvas-contract types (`CanvasItem`/`CanvasProps`/`CanvasComponent`) moved here from `canvas-registry.ts`, which is **deleted** (folded in).
2. **Core is the first module (`coreModule`).** The five system types (task/meeting/note/link/entity) are a built-in manifest — all markdown-canonical, only `link` declaring a bespoke `canvasId`. So the **whole app now resolves type behavior through this one boundary**; the default markdown experience is byte-for-byte unchanged. This is the strongest proof the boundary is real, and the exact shape Papers/Songs follow.
3. **`src/lib/module-wiring.tsx` — the impure wiring half.** The one thing that can't be pure: `canvasId → React component` (`markdown`→`MarkdownCanvas`, `link`→`LinkCanvas`) + `canvasComponentFor(id)` with a `?? MarkdownCanvas` fallback. Linked to the policy by the `canvasId` string. So a verify script (or any pure import of the registry) never drags the editor bundle in. `ItemCanvas` now does `canvasComponentFor(canvasIdForType(item.type, owner.id))` — owner-aware so the per-user enable flip routes a disabled module's type back to the default without touching the call site.
4. **Exporters are deterministic code on the def.** `ExporterDef.render` is plain code (Principle 7, no model). A real module with a heavy renderer (pandoc→docx) keeps that in its own module file, which core's pure registry never imports, so core stays node-pure. (The OneDrive export *engine* is core infrastructure, not one of these per-type module exporters.)
5. **Canonical-body-format-per-type is now a real platform capability** (answers Tyler PR #1 decision #1 in code): `canonicalFormatForType(type)` resolves the owning type's declared format (markdown by default; a type may declare `chordpro` etc.). No renderer/editor rewiring yet — everything is still markdown — but the slot Songs/Papers read from exists and resolves.
6. **Per-user enable seam (the "later flip").** `isModuleEnabled(moduleId, ownerId?)` returns the manifest default today and ignores `ownerId`; per-user enablement becomes a lookup against a settings table right here, no call-site change. A disabled module's type resolves to the default canvas, contributes no exporters, and reports no format override.
7. **Proof = a reference registration, not a module (ADR-042).** `referenceModule` is exported from `modules.ts` as executable documentation touching all four slots (a type with its own format + canvas, an exporter whose `render` runs, a pull integration); it is **not** in the built-in list, so the live app ships core-only and no DB `types` row exists for it. The M6 verify script registers it (plus a ChordPro-format fixture and a disabled fixture) to prove resolution; core's live `link`→`LinkCanvas` proves the enabled canvas-wiring path end-to-end.
**Why / alternatives:** *Keep the M5 split (policy pure, wiring impure), generalized to all four slots, linked by string ids* — chosen; it preserves node-testability and matches what M5 called "M6-shaped, not throwaway." *One combined registry returning components* — rejected: importing it drags the editor bundle into the verify script (the exact thing M5 avoided). *Make core a special case outside the registry* — rejected: registering core as the first module means one boundary the whole app flows through, the honest dogfood. *Ship a live reference module/type* — rejected per ADR-042 (foundation ships the capability, not modules); the reference registration lives as exported documentation + a verify fixture, zero live footprint. *Build the per-user enable storage now* — rejected as out of scope; the seam (owner-threaded resolvers, `isModuleEnabled`) is in place so it's a later table lookup, not a refactor. *Move label/icon authority into the registry* — rejected: the DB `types` table stays the runtime source for label/icon enumeration (it's per-owner data); the registry owns the type's *code behavior* (canvas/format/exporters), and a module's manifest *declares* label/icon only for its future install/seed step.
**Verification:** `tsc --noEmit` clean; full `next build` clean (✓ compiled, all routes incl. `/items/[id]`, `/items/[id]/print`, `/share/[token]`; ran with the dev stand-in's set-but-empty Clerk keys). New **`scripts/verify-module-registry.mts` 38/38** — core dogfooding (5 types resolve, `link` keeps its canvas, no core exporters, all type keys enumerated), a workflow module slotting in (register `referenceModule` → its canvas/format/exporter[run deterministically]/integration all resolve; unknown before registration), format-per-type (a ChordPro fixture resolves `chordpro` + its chord canvas), the enable seam (a disabled fixture's type → default canvas, no exporters, no format override, not enumerated), and duplicate-id rejection. **`scripts/verify-canvas-seam.mts` still 16/16** (repointed to `modules.ts`; the M5 invariants + the ADR-018 footer-fields invariant unchanged). **In-browser (dev stand-in, auth-stub):** the `link` item renders the URL chip above the title with the full default canvas beneath (toolbar, URL strip, Related, Save Offline, Share, Fields) — core's bespoke canvas resolved through the boundary; a `note` renders the unchanged default markdown canvas (no chip, headings/emphasis/rule intact) — zero console warnings/errors, zero server errors. No schema/DB change, no new dependency.
**Affects:** new `src/lib/modules.ts`, `src/lib/module-wiring.tsx`, `scripts/verify-module-registry.mts`; edited `src/components/canvas/ItemCanvas.tsx` (resolve via the boundary, owner-aware), `src/components/canvas/MarkdownCanvas.tsx` + `LinkCanvas.tsx` (`CanvasProps` import → `@/lib/modules`), `scripts/verify-canvas-seam.mts` (import → `@/lib/modules`); **deleted** `src/lib/canvas-registry.ts`. **Phase M (the Markdown-epoch foundation rework) is complete** — M1–M3 (ADR-038/040), M5 (ADR-041), M6 (this). The modules themselves (Papers, Songs, …) are post-foundation, non-core work on top of the capability (ADR-042; mostly Tyler's lane).

## ADR-045: Slice 34 — per-type item templates (Phase 3, Tier 2a)
**Date:** 2026-06-13
**Status:** accepted. Touches **core** (a new `templates` table — the data model is a both-agree area per CLAUDE.md "Building together"). **Brandon directed shipping Tier 2 solo** (same waiver as ADR-043/044 — "do whatever is logical and safe, then commit"); flagged to Tyler in `COLLAB.md`. First half of roadmap **Phase 3, Tier 2 — templates** (the workflow/wiki structure generator is ADR-046, the other half).
**Context:** Tier 1 (ADR-044) gave types a `property_schema`; Tier 2 makes *starting from a preset* cheap. Per-type item templates store property defaults + starter body content (roadmap Phase 3, Brandon 2026-06-12). The Phase-2 meeting-prep agenda (`src/lib/meetings/prep.ts`, `DEFAULT_AGENDA`) is the named forerunner: a hardcoded starter body for one type; this generalizes it so any type carries named, editable starting points, authored in the same editor items use.
**Decision:**
1. **New `templates` table** (migration **0009**): `{id, owner_id → users, type → types.key (ON DELETE cascade), name, body jsonb, property_defaults jsonb, created_at}`, indexed `(owner_id, type)`. **Owner-scoped** (a template is personal config, like a `views` row — not shared structure), unlike the instance-global `types` registry. A table, **not** `items` (rule 2 covers user *content*; a template is config, the same call `views`/`matchers` made) and **not** a column on `types` (templates are many-per-type, owner-scoped, and a starter body on the hot `getType` read — called on every canvas mount — would fatten it; a side table keeps type reads lean). The cascade drops a type's templates if the type itself is deleted (`deleteType` already blocks while *items* use it).
2. **`src/lib/templates.ts` store** (mirrors `views.ts`/`types.ts`: hand-rolled validation, one parse path, row→definition coercion). `body` accepts a raw markdown string (wrapped to `{format,text}`) or an already-shaped body; an empty body stores null. `parseDefaults` drops null/empty values so a default never writes an empty key. `type` is set on create and **immutable on patch** (the defaults are keyed to that type's schema; re-pointing would orphan them). CRUD + `createItemFromTemplate` + `templateCountsByType`.
3. **`createItemFromTemplate` = `createItem` with the body + defaults filled in**, `inbox: false` (deliberate creation, like "+ New" — *not* an Inbox arrival, ADR-010) and a blank title (the user names it). So a template rides the whole existing item-create path (revisions, mention-sync, body-text/FTS) for free.
4. **API:** `GET/POST /api/templates` (+ `?type=` filter), `GET/PATCH/DELETE /api/templates/[id]`, and `POST /api/templates/[id]/apply` — a distinct endpoint so "create from template" is explicit and the store owns the seeding (rather than overloading `POST /api/items`).
5. **Builder UI** (`/build/templates{,/new,/[id]/edit}` + `TemplateBuilder.tsx`): name, type select (immutable when editing), a **controlled property-defaults editor** keyed off the selected type's `property_schema` (the mirror of `CustomProperties`, which edits a live item), and the **starter body in the same Tiptap editor items use** — reused in *controlled* mode (`LazyMarkdownEditor` with a stable `initialMarkdown` + `onChange`, no item to autosave against; `itemId` is optional, only for @-mention self-exclusion). `/build` home gains a real "Item templates" card.
6. **Template-aware "+ New"** (`NewItemButton`): self-fetches `/api/templates?type=` on mount and, when the type has templates, becomes a menu — "Blank" + each template. Done in the *one* component, so every list page that already renders `<NewItemButton type=…/>` (the five system lists + `/list/[type]`) gets the menu with **zero page changes**.
7. **The meeting-prep forerunner is now an expressible instance** (not rewired): `DEFAULT_AGENDA` could be saved as a `meeting` template; prep still synthesizes its panel live (ADR-025). Generalizing prep onto templates is a later, optional step.
**Why / alternatives:** *Templates as `items`* — rejected (would pollute every owner-scoped list/search/FTS/export unless excluded everywhere; config belongs in a side table). *Templates as a `types` column* — rejected (many-per-type + owner-scoped + bloats the hot `getType`). *Mutable template type* — rejected (orphans the defaults). *Seed hot fields like due/urgency into templates* — deferred; the contract is type-uniform (`body` + `propertyDefaults`) for now, and a relative-due offset is the more useful future shape than an absolute date. *Title default* — deferred (blank, like "+ New"). *Overload `POST /api/items` for apply* — rejected for the explicit `/apply` endpoint.
**Verification:** `tsc --noEmit` clean; full `next build` clean (new routes `/api/templates{,/[id],/[id]/apply}`, `/build/templates{,/new,/[id]/edit}`, ran with the dev stand-in's empty Clerk keys). New **`scripts/verify-templates.mts` 27/27** against Neon under throwaway owners — parse/validate (name/type required, body string→wrapped, empty→null, defaults drop empties, array rejected), CRUD, owner-scoping (foreign get/update/delete are not-found), apply (seeds body + defaults, `inbox` false, blank title), and the type cascade. **In-browser (dev stand-in):** built an "Interview candidate" template for a custom type through the real form (the markdown editor mounted in controlled mode, the property-defaults section rendered the type's fields, a URL default set); it landed on the `/build/templates` index grouped by type with a "1 field" badge; the type's list showed **"+ New ▾"** with `["Blank","Interview candidate"]`; applying it created an item carrying `{website_link: …}` with `inbox:false` and a blank title — zero console errors. Verification template + item were cleaned up.
**Affects:** `src/db/schema.ts` (+`templates`), `drizzle/0009_*`; new `src/lib/templates.ts`, `scripts/verify-templates.mts`, `src/app/api/templates/route.ts` + `[id]/route.ts` + `[id]/apply/route.ts`, `src/components/build/TemplateBuilder.tsx`, `src/app/build/templates/page.tsx` + `new/page.tsx` + `[id]/edit/page.tsx`; edited `src/components/home/NewItemButton.tsx` (template menu) and `src/app/build/page.tsx` (Item templates card). No new dependency.

## ADR-047: Slice 36 — the MCP server (Phase 3, Tier 3): a hand-rolled Streamable-HTTP endpoint over the existing item libs
**Date:** 2026-06-13
**Status:** accepted. **CORE** — the machine/MCP API contract is on the both-agree+ADR list (CLAUDE.md "Building together"). **Shipped solo per Brandon's standing Tier-direction** ("build the next tier"), same waiver as M6 / slices 33–35; flagged to Tyler in `COLLAB.md`. Opens roadmap **Phase 3, Tier 3 — the Claude layer** (the MCP server; scheduled Claude tasks + Notion migration follow).
**Context:** PRD §5.5 + Principle 4 ("Claude is a first-class client"): a thin MCP server exposing search/read/create/update items + list-by-entity/date, auth via a personal API token, enabling "what's open with Roger," "file this as a task due Friday," "prep tomorrow's 1:1" from Claude desktop/web/mobile. The two judgment calls were *how to speak MCP* (SDK vs hand-roll, and where it's hosted) and *how the token resolves to an owner* (the token scheme stores no owner).
**Decision:**
1. **Hand-rolled MCP over the Streamable HTTP transport, no SDK.** For request/response tools the protocol is a small, stable JSON-RPC surface (`initialize`, `tools/list`, `tools/call`, `ping`), so we implement it directly rather than take `@modelcontextprotocol/sdk` — the same dependency call ADR-034 made for Web Push (Principle 5). The protocol layer is verified against the **2025-06-18 spec** (lifecycle + transports): a POST carrying a JSON-RPC *request* gets one `application/json` response; a POST carrying only a *notification* (e.g. `notifications/initialized`) gets **202, no body**; GET/DELETE → **405** (we offer no server-initiated SSE stream and keep no session). Version negotiation echoes the client's version when supported (we accept 2024-11-05 / 2025-03-26 / 2025-06-18), else returns our latest.
2. **In-app endpoint at `POST /api/mcp`, stateless.** No separate process to host or keep warm (Principle 4 $0, Sunday-proof) — it's a Next route. **Stateless** (no `Mcp-Session-Id`): every call re-authenticates and re-resolves the owner, so there's no per-connection state, which suits a serverless function and a single user. Added to the `proxy.ts` public set (Clerk never runs on it); the machine token is the only credential, which also makes the spec's DNS-rebinding `Origin` check moot (a forged origin can't forge the bearer).
3. **Auth = the existing machine-token scheme (ADR-004) with a new `mcp` scope.** No change to the token format; `verifyMachineToken(authz, "mcp")` gates the route. Owner resolution mirrors the other machine jobs: `resolveMcpOwner()` reads `LEDGR_MCP_OWNER_UPN || ONEDRIVE_EXPORT_UPN || GRAPH_MAILBOX_UPN || DEV_USER_EMAIL` → the `users` row (single-user v1; a multi-user build would map each token to its own owner).
4. **Six tools, each a thin wrapper over the libs the REST API already uses** (search.ts / views.ts / items.ts / relations.ts / types.ts), so the MCP surface can't drift from the app's own contract or skip owner-scoping: `search_items` (FTS), `list_items` (by type/status/kind/due-window/entity — the "list by entity/date"), `get_item` (fields + markdown body + relations), `create_item` (body as markdown, `relateTo` links to existing items, defaults `inbox:false`), `update_item`, and `list_types` (so the model learns the type/property vocabulary). `create_item`/`update_item` **reuse `parseItemPayload`**, so an MCP write validates exactly like an `/api/items` write.
5. **Pure/impure split, the M5/M6 discipline.** `src/lib/mcp/protocol.ts` (envelope, version negotiation, message classification — pure, node-testable), `server.ts` (the method dispatcher), `tools.ts` (the registry + `callTool`), `owner.ts` (resolution); the route is a thin transport shell. **`parseItemPayload`/`asUuid` were extracted from `api.ts` into a new pure `src/lib/item-input.ts`** (api.ts re-exports them, so the 17 route call sites are untouched) — `api.ts` imports `next/server` + the Clerk owner chain, which a node verify script can't load, so the parsing had to move to stay reusable.
6. **`/health` `checks.mcp`** canary (`hasScopedToken("mcp")` + `resolveMcpOwner()`), like `checks.graph` — never changes overall status. Tool errors surface as MCP `isError` results (the session survives); unexpected errors are `captureError`-ed (source `mcp`) with a correlation id.

## ADR-048: Papers module — tabbed canvas, scaffold-in-properties, docx-via-route
**Date:** 2026-06-14
**Status:** accepted
**Context:** The second workflow module in Tyler's lane (after Songs), built on the M5/M6 foundation (ADR-042/043). A `paper` is a seminary-paper workspace replacing the hand-run "HTML outline viewer + MSM Word doc" flow (`ty-docs/Paper_Outline_Viewer_Build_Spec.md`). Three parts the writer builds in order: a **quote bank** (sources gathered before drafting), an **outline** (sections/paragraphs with notes), and the **final paper**. A working MSM `.docx` renderer already existed as a CLI (`ty-docs/msm-render.js`). Module internals are "not core / move fast solo" per CLAUDE.md; the one core-adjacent temptation (a binary exporter) was deliberately avoided.
**Decision:**
1. **One `paper` item, tabbed canvas — Quote Bank · Outline · Draft** (`PaperCanvas` → `PaperCanvasClient`, wired as canvas id `paper`). The **Draft** is the canonical markdown `body` (the deliverable; gets revisions + FTS; exports to docx). The **Outline** (markdown) and **Quote Bank** (structured) are the *scaffold* that sits beside it. (Tyler's model: outline its own markdown surface, quote bank alongside it, both together and separate from the final draft.)
2. **Scaffold lives in `items.properties`** — `properties.outline` (markdown string) + `properties.quoteBank` (`QuoteEntry[]`) + title-page meta keys. No schema migration: the existing `PATCH /api/items/:id` already replaces `properties` wholesale. **`PaperCanvasClient` is the single writer of a paper's properties** (it preserves unknown/system keys and overrides only the keys it owns), so `PaperCanvas` does **not** render the generic `CustomProperties` panel — two whole-object properties writers would race. The title-page meta form (`PaperMeta`) lives inside the Draft tab.
3. **Deterministic MSM citation engine** (`src/lib/papers/citation.ts`, pure): structured source (book | video) + page → the three MSM 4th-ed footnote forms (Full / Short / Ibid.), transcribed from the build spec. The quote bank shows all three per entry; **Cite** mints a unique `[^id]` marker, splices it at the Draft caret, and appends the matching `[^id]: …` definition — the markdown-canonical footnote mechanism the renderer consumes. Each insertion gets a fresh marker so a first reference and a later shortened one are distinct numbered notes.
4. **MSM `.docx` ships via a dedicated route, not the module exporter slot.** `ExporterDef.render` returns a `string`; a `.docx` is a binary buffer. Rather than widen that contract (a **core** change needing Brandon + ADR), the renderer is ported to `src/lib/papers/msm-docx.ts` and served by `GET /api/items/[id]/render-docx` (owner-scoped, `paper`-only). `paperModule.exporters` is empty. The positional `FootnoteReferenceRun` approach makes the CLI's old duplicate-id unpack/pack fix unnecessary.
5. **Draft + Outline use a bespoke raw-markdown textarea** (`PaperMarkdownArea`), not the shared Tiptap WYSIWYG. Papers are footnote-heavy academic prose; `[^id]` footnotes are a raw-markdown feature Tiptap neither renders nor exposes an insertion handle for. The textarea makes footnotes first-class and click-to-cite trivial, and keeps Papers self-contained (no edit to the shared editor). CodeMirror is a future upgrade (the scoped plan's "textarea → CodeMirror later").
6. **Type seeded with a `property_schema`** (title-page meta as text fields + `stage` select) via `scripts/seed-papers.mjs` (idempotent, `is_system=false`). The schema is the data-model declaration; the canvas reads/writes these keys directly. `PAPER_STAGES` source of truth is `src/lib/papers/types.ts`.
**Why / alternatives:** *Outline-generates-the-markdown-skeleton* — rejected (lossy round-trip; notes leak into the deliverable). *Outline/quote bank as separate child items* — deferred (heavier cross-item orchestration; the caveat below is the upgrade path if `properties` bloats). *Widen `ExporterDef` for binary* — rejected (core change for one module's need; a route is cleaner). *Reuse the Tiptap editor for the Draft* — rejected (no footnote node, no external insertion handle; would require editing a shared core component). *pandoc for docx* — rejected (system binary, not Vercel-friendly; the JS `docx` package is proven in `ty-docs`).
**Caveat (logged):** `items.properties` is selected on normal item reads, so a very large outline/quote bank bloats those payloads (Principle 8). Acceptable for v1 (single user, modest paper counts); promote the outline to a child `note` item if it grows (gets its own revisions/FTS).
**New dependency:** `docx` (runtime) — the only realistic MSM-compliant Word output; pandoc needs a binary. Single new runtime dep.
**Shared-file touches (flag for Brandon):** `src/lib/modules/register.ts` (+`paperModule`) and `src/lib/module-wiring.tsx` (+`paper` canvas) — both additive; the five core types are unchanged.
**Verification:** `scripts/verify-papers.mts` **16/16** (registration/resolvers, citation engine book+video Full/Short/Ibid, a real `renderMsmDocx` → 10KB buffer with 3 positionally-allocated footnotes). Regression green: `verify-module-registry` 38, `verify-canvas-seam` 16, `verify-songs` all pass. `tsc` + `next build` clean (`/api/items/[id]/render-docx` registered); eslint clean; `paper` type seeded to the dev DB. (In-browser canvas walkthrough left for Tyler's morning check.)
**Affects:** new `src/lib/papers/{types,citation,msm-docx}.ts`, `src/lib/modules/papers.ts`, `src/components/canvas/PaperCanvas.tsx`, `src/components/paper-editor/{PaperCanvasClient,PaperMarkdownArea,QuoteBank,PaperMeta}.tsx`, `src/app/api/items/[id]/render-docx/route.ts`, `scripts/{seed-papers.mjs,verify-papers.mts}`, `explorations/song-import.md`; edited `src/lib/modules/register.ts`, `src/lib/module-wiring.tsx`, `package.json` (+`docx`).

## ADR-052: Slice 37 — the weekly health check (Phase 3, Tier 3b): a deterministic scheduled self-monitor
**Date:** 2026-06-14
**Status:** accepted. **Not core** — a scheduled cron over existing canaries + the existing Web Push sender, no data-model, body-format, or MCP-contract change, so it's "move fast, solo." Shipped solo per Brandon's standing Tier direction ("build the next tier"); flagged to Tyler in `COLLAB.md`. Advances roadmap **Phase 3, Tier 3 — the Claude layer** (the second of its three items: MCP server done in ADR-047, this is "scheduled Claude tasks," the selective Notion migration remains). **Renumbered on merge:** authored on the `claude/next-tier-slices-opeeao` branch as ADR-048; that number was taken on `main` by the Papers module (and 049/050/051 followed), so it became ADR-052 when the branch merged in (2026-06-14).
**Context:** PRD §5.5 lists "scheduled Claude tasks (morning briefing, weekly health check)" and §6.2 spells the health one out: "a weekly scheduled Claude task hits `/health` and emails Brandon on failure." The morning-briefing half already exists in its deterministic form — the daily agenda push (slice 30, ADR-034) is a count summary of the day. This slice builds the missing sibling, the weekly health check: the "no silent failures" guard (Principle 9) and specifically the §12 failure mode where GitHub Actions auto-disables after 60 days of repo inactivity and a sync stalls unnoticed.
**The Principle-3 tension, resolved deterministically.** Principle 3 says "AI lives only in the deliberate, human-in-the-loop Claude/MCP layer, **never in a cron job**." A literal "scheduled *Claude* task" that calls the Anthropic API on a timer to author prose would violate that — and amending a principle is a both-agree+ADR core decision, not a solo call. So this job is **plain code, no model**: it reads canaries and applies fixed rules. The richer AI-authored briefing the PRD imagines stays human-in-the-loop (ask Claude over the MCP server, ADR-047, which can already read everything). If Brandon later wants model-authored scheduled output, that's a separate, principle-amending decision (and a line on the $0 cost target) — noted, not taken here.
**Decision:**
1. **`gatherHealth()` extracted to `src/lib/health.ts`.** The `/health` route's snapshot logic (DB check, every job canary, MCP/Graph, captured errors) moved into one function returning a typed `HealthReport`; the route is now a thin wrapper. So the self-monitor reads the **exact same canaries** in-process rather than self-calling its HTTP endpoint — cleaner than §6.2's "hits `/health`" phrasing, and it still works when routing itself is the problem. The JSON shape is unchanged (additive: a new `checks.healthCheck`).
2. **A pure `evaluateHealth(report, recentErrorCount, now)` → `HealthAlert[]`** (in `src/lib/health-check.ts`, node-testable, the M5/M6 discipline). Fixed, deterministic rules: **DB down** → critical; **captured errors** over a 7-day window → warn; a **configured-and-failing Graph grant** → warn (the secret-expiry alarm); a **stalled scheduled job** → warn. Alerts sort critical-first so the push leads with what matters.
3. **"Stalled, not merely unconfigured."** A freshness rule fires only when a job *ran before* (its `…RunAt` canary is set) but its last clean success is older than a per-job budget (export 48h, calendar/todoist 24h, email 12h, agenda 48h). A job that never ran is a Brandon-step, not a fault, and stays silent — the alpha posture and Sunday-proofing both say don't cry wolf.
4. **`runHealthCheck(ownerId, sender, opts)`** gathers (or takes an injected report, for tests), evaluates, and on a non-empty alert list **pushes once** via the existing `PushSender`/`sendToOwner` (slice 30) and stamps `job_state` `health:check` with `lastAlertAt` + the alerts; on a clean run it stamps `lastSuccessAt` and stays **silent** (PRD §6.2 is failure-only). The sender may be **null** (VAPID unset): the job still evaluates and records — the value is the self-monitoring, not only the notification. **Findings are written to `job_state`, never `error_log`**, so rule-2 can't feed back on itself.
5. **Delivery is Web Push, not email.** §6.2 says "emails Brandon," but `email-out` isn't built (parked, §5.4) and push is the channel Ledgr already has. `checks.healthCheck` in `/health` is the durable record either way.
6. **`GET /api/machine/health-check`** (cron-scoped machine token) + **`.github/workflows/health-check.yml`** (weekly Mon 13:00 UTC). Weekly rides the GitHub Actions scheduler seam like notify-prep — the same `LEDGR_CRON_TOKEN`, swappable for a local cron in Phase 4. A non-200 fails the workflow loudly (GitHub emails) — a second, transport-level signal independent of the app.
**Why / alternatives:** *Call `/health` over HTTP from the job* (§6.2's literal phrasing) — rejected for an in-process `gatherHealth()` (no self-HTTP/token juggling; works when routing is the fault). *Write alerts to `error_log`* — rejected (it'd feed the "captured errors" rule a week later, a false-alarm loop); `job_state` + the canary instead. *Send a weekly "all green" push* — rejected (notification spam; failure-only per §6.2). *Alert on any unconfigured integration* — rejected (everything is Brandon-step-blocked in alpha; only a *previously-working, now-stalled* job alerts). *Model-authored briefing* — deferred; would amend Principle 3 (core, both-agree) and the cost target.
**Verification:** `tsc --noEmit` clean (re-run after merge into current `main`). The pure evaluator proven 15/15 against synthetic snapshots (green→no alerts; DB-down→critical; errors→warn with window naming; Graph configured-failing alerts while unconfigured stays quiet; stalled-vs-never-ran-vs-fresh; per-budget; critical-sorts-first). `scripts/verify-health-check.mts` carries the DB-backed half (an unhealthy injected report pushes + stamps `lastAlertAt`; a healthy one is silent + stamps `lastSuccessAt`; a null sender still records). `next build` is the Vercel production gate (dev server up locally). **Brandon-step to light it up:** none new — the weekly workflow uses the existing `LEDGR_CRON_TOKEN`, and delivery uses the existing VAPID keys (§1e); until VAPID is set the check still runs and records to `/health`, it just can't push.
**Affects:** new `src/lib/health.ts` (extracted `gatherHealth` + `HealthReport`), `src/lib/health-check.ts`, `src/app/api/machine/health-check/route.ts`, `.github/workflows/health-check.yml`, `scripts/verify-health-check.mts`; edited `src/app/health/route.ts` (thin wrapper + the `healthCheck` canary), `runbook.md` (§2 canary, §2a job table). No new dependency, no schema/DB change.

## ADR-057: In-context delete confirmation as the standard; type-delete can take its items
**Date:** 2026-06-15
**Status:** accepted (Tyler's call). The confirmation-UI half is **not core** (presentation). The **type-delete cascade touches core** (item-deletion semantics over the `items`/`types` data model) — shipped solo at Tyler's direction under the same waiver pattern as the M6/slice-33 core work, but flagged to Brandon in `COLLAB.md` as a core-touch to ratify. **The hard-delete in decision point 2 below is superseded by ADR-058** (type-delete now soft-deletes to Trash); the in-context-confirm half (point 1) stands.
**Context:** Five destructive actions used the browser `window.confirm()` popup (type/view/template delete, collab-notes clear). **Decision:**
1. **`ConfirmButton` (`src/components/ui/ConfirmButton.tsx`) is the new standard** for any confirm-before-delete: a trigger that opens an in-context popover (title/description, optional extra body, Confirm/Cancel) anchored to itself, closing on outside-click/Esc. `onConfirm` may throw to surface an API error inside the popover. Checked, it calls `DELETE /api/types/[key]?withItems=1` → `deleteTypeWithItems` (`src/lib/types.ts`), which **hard-deletes** the owner's items of that type plus their descendants (recursive on `parent_id`) in one statement, then drops the type. Relations/revisions/attachments cascade (existing FK `ON DELETE cascade`); the type's templates cascade with the type row.
**Why / alternatives:** Hard-delete (not soft-delete to Trash) because a trashed row still carries `type = key` and would keep the FK blocking the type delete — and deleting a type is itself a permanent Build action, so the items go with it. Owner-scoped (the owner-scope-every-query invariant; one user per deploy), with a global re-check before dropping the type row so a cross-owner reference surfaces a clear message rather than a raw FK error. `parent_id` is `NO ACTION` (not `RESTRICT`), so a single statement removing a whole subtree is safe. The UI is explicit that the items are removed permanently, not trashed.
**Affects:** new `src/components/ui/ConfirmButton.tsx`; `src/lib/types.ts` (+`countItemsOfType`, `deleteTypeWithItems`); `src/app/api/types/[key]/route.ts` (`?withItems=1`); `src/app/build/types/[key]/edit/page.tsx` (+`itemCount`); `TypeBuilder.tsx`/`ViewBuilder.tsx`/`TemplateBuilder.tsx`/`CollabNotes.tsx` (confirm → `ConfirmButton`). `scripts/verify-nav-and-deletes.mts` covers the cascade against Neon.

## ADR-058: Types are soft-deleted to Trash (recoverable), not hard-deleted — supersedes ADR-057's hard-delete
**Date:** 2026-06-15
**Status:** accepted (Tyler's call, the same session). **Core-touch** (extends the soft-delete model to the `types` table) — flagged to Brandon in `COLLAB.md` to ratify, like ADR-057. Supersedes decision point 2 of ADR-057.
**Context:** ADR-057 hard-deleted a type's items when deleting the type. That quietly broke the standing **"Soft-delete only. Deletes move to Trash (30-day purge)"** convention (CLAUDE.md) — deleting a type was the one destructive action with no undo. The fix is to bring types into the same Trash model.
**Decision:**
1. **`types` gains a `deleted_at` column** (migration `0014_nice_fantastic_four.sql`). A soft-deleted type stays as a row (so its trashed items keep a valid `items.type → types.key` FK) but drops out of `listTypes()` (registry, pickers, Build list, quick-capture). `getType` still resolves it so labels render for trashed items; `assertTypeExists` excludes it so you can't create/retype an item into a trashed type.
2. **Deleting a type soft-deletes it.** Empty type → `deleteType` stamps `deleted_at`. In-use type via `?withItems=1` → `softDeleteTypeWithItems` stamps the type and its live items (+ descendants, recursive on `parent_id`) with **one shared `now()`** so they form a restorable unit. Items go to Trash, not hard-deleted; relations/revisions/attachments are left intact for a clean restore.
3. **Restore.** `restoreType` (`POST /api/types/[key]/restore`) revives the type and the items that share its `deleted_at`. Restoring a single item from Trash (`restoreItem`) auto-revives its type if that type is soft-deleted — an active item never references a hidden type. The Trash page gained a "Types" section (`listDeletedTypes` + `TypeRestoreButton`) listing soft-deleted types with their trashed-item count.
4. **Purge.** `purgeExpiredTrash` hard-deletes soft-deleted types past the retention window after it purges items (the FK is clear by then; the type's templates cascade with the row).
**Why / alternatives:** Soft-deleting only the type while leaving its items live was rejected — it would orphan live items under a hidden type. Surfacing types as first-class Trash entries (vs. only implicit recovery by restoring an item) was chosen because the user explicitly wanted "the type and the things associated with it" both recoverable. The Trash "N items" label counts the type's own trashed items; restore also brings back their descendants (a deliberate, documented small mismatch).
**Affects:** `src/db/schema.ts` + migration `0014`; `src/lib/types.ts` (`deletedAt` on `TypeDefinition`, `listTypes` filter, `deleteType` soft, `softDeleteTypeWithItems` (was `deleteTypeWithItems`), `restoreType`, `listDeletedTypes`, `countLiveItemsOfType`); `src/lib/items.ts` (`assertTypeExists` excludes deleted, `restoreItem` revives type, `purgeExpiredTrash` purges types); `src/app/api/types/[key]/route.ts` + new `restore/route.ts`; `src/app/trash/page.tsx` + new `src/components/trash/TypeRestoreButton.tsx`; `TypeBuilder.tsx` copy. `scripts/verify-nav-and-deletes.mts` 38/38 (soft-delete, hide-from-registry, restore, auto-revive, create-blocked). **Coordination:** other instances run `npm run db:migrate` after pulling (additive nullable column, safe).

## ADR-060: Pre-built Bible/passage relational database — verse is the atomic unit
**Date:** 2026-06-14 (Brandon + Tyler build meeting; logged 2026-06-15)
**Status:** accepted. **Core** — a data-model/relational decision (the `passage` relational hub, how references resolve into the relation graph). Both agreed in the 6.14 session. Adopts and closes `explorations/scripture-passages-as-entities.md`. The module *build* (the table seed, the auto-tagger) is non-core module work and lives in the build queue, not the foundational rework. *(Numbered 060 because the same-evening entity→person retirement and the nav/type batch took 055–059.)*
**Context:** A long debate over how scripture references should connect to content. Tyler argued for **pre-shipping** the whole Bible as a relational structure so every reference auto-connects; Brandon initially argued free-typed tags (Notion-style "type a tag, press Enter to create") already covered it. The resolution went to pre-built: a free-typed tag does not guarantee correct *shape* and only tags *one side* of the relation, so you'd still have to visit every other item to tag it consistently ("did I tag it with generosity or giving?"). A pre-built relational table removes the two-sided tagging burden entirely and makes "show me everything I have on Hebrews 11:1" a graph query, not a tag-hygiene exercise.
**Decision:**
1. **Ship the Bible as a pre-built relational structure: book → chapter → verse, with the verse as the atomic unit.** A chapter is the container of its verses; a single-verse reference and a multi-verse passage (e.g. Luke 11:35-36) both resolve through the same structure, with a **passage range linking each individual verse in the range**. Sub-verse granularity (11:35a/b) is explicitly **rejected** as too far.
2. **A deterministic, RefTagger-style auto-tagger** recognizes reference patterns in many surface forms ("Luke 11:35", "LK 11.35", "Luke 1135") and wires the relation automatically. **It is plain code / a cron job, no model in the loop** (Principle 3). Any remaining representation edge cases get resolved with Claude's help during the build, not at runtime.
3. **Modeled as a bespoke `passage` type, related to via the existing `relations` table** — not as a `kind` of a generic "entity." Note (ADR-055): the `entity` meta-type was retired and the `kind` column dropped, so the original "passage as an `entity.kind`" framing in the exploration is superseded; a passage is its own system/bespoke type, a verse is an item, and any other item relates to it through `relations`. This does **not** need the deferred `relation` property kind (item-to-item relations already exist) — the same finding that let ADR-055 drop `entity`.
**Why pre-built over tag-on-demand:** the relational table does the work once for everyone; content connects without retroactive tagging of existing items, and there is no tag-inconsistency drift. Storage cost is negligible (Markdown + relations).
**Affects:** `schema.md` (a `passage` type + its book/chapter/verse rows + relations; forthcoming), the relations engine, a new bespoke Bible/passage module + auto-tagger (build queue). Feeds future passage-organized inputs (Quiet Time Journal API, Logos).

It settles the organize-in-app half of `storage-organization.md`; the export-to-MD-file-tree mapping (one primary parent → folder path, the rest → wiki-links) remains the open part of that exploration.
3. **A "project" is not a functional type** — it is a relational connection across tasks/meetings/notes. Confirmed against live test data: nothing functionally changes about a task when it gains subtasks or a parent project; it just needs relations. Confirms `explorations/project-items.md`.
**Affects:** the notes module design, the type/canvas model docs, `explorations/storage-organization.md` and `project-items.md` (org half resolved here).

## ADR-064: Customizable dashboards — multiple named dashboards, widgets as jsonb, dashboard-level focus
**Date:** 2026-06-15
**Status:** accepted (Tyler's lane — views/dashboards are explicitly non-core per CLAUDE.md). A new `dashboards` table is an additive schema change, not a touch to core `items`/`types`/`relations`. From the approved plan `~/.claude/plans/ok-let-s-quick-build-crispy-neumann.md` (interview with Brandon). Supersedes the single-dashboard model of ADR-031.
**Context:** ADR-031's dashboard was one per owner: pinned views rendered as uniform 8-item list previews, ordered by a single `views.dashboard_order` integer — no per-widget settings, no layout-faithful rendering, no multiple dashboards. Brandon wanted multiple named dashboards (Home, Today, a project-focused one), each a freely resizable/draggable grid of widgets, where a widget is a "chunk" of a view with its own gear settings, plus a dashboard-level "focus" that scopes every widget to one project at once.
**Decision (Slice 1 — the data/API foundation):**
1. **New `dashboards` table** (migration 0016), one row per dashboard, with the full widget array inline as a `widgets` jsonb column — matching how `views`/`templates` store config as jsonb and keeping a dashboard read to one row + a batched per-widget fan-out (no normalized `dashboard_widgets` table, no join, atomic layout saves). Columns: `name`, `position`, `focus_item_id` (FK items, **ON DELETE SET NULL** so deleting the focus clears it without dropping the dashboard), `widgets`.
2. **Three widget kinds:** `view` (renders a chunk of a saved view), `stat` (a count from a view's filter), `action` (non-data: quick-capture / new-from-template / link). `view`/`stat` require a real `viewId`; `action` has none. **Every data widget is backed by a real saved View** (Brandon's call — no hidden/ad-hoc views).
3. **Widget settings are display-only overrides** (itemLimit, sortOverride, renderStyle compact|faithful, titleOverride for view; label for stat; action config) applied in-memory at render/query time — they **never mutate the backing view** (asserted in the verify script).
4. **Dashboard-level focus** is a pure filter-merge: `applyFocus(filter, focusItemId)` injects `relatedTo=focusItemId` when focus is set and the widget doesn't already pin its own relation. `viewWhere` already restricts `relatedTo` to confirmed edges, so there's no new query logic.
5. **Home/Today assignment** rides `users.settings` (no schema change): nullable `homeDashboardId`/`todayDashboardId`, tolerant-parsed back to null if the id is malformed/removed, so a deleted dashboard silently falls back to the fixed built-in layout. (The `/` and Today route resolution lands in Slice 7.)
6. **`usedViewIds(ownerId)`** scans dashboards' widgets jsonb for placed view ids — powers the Views page's "in use as a widget" badge + filter (Slice 6) cheaply at one-owner scale, no FK/join.
7. **`?count=1` on `GET /api/items/query`** returns just `{ count }` so focus-aware stat widgets stay one cheap call on client refetch.
**Sequencing note:** Slice 1 is **purely additive** — the old single-dashboard model (`views.dashboard_order`, `pinView`/`unpinView`/`setDashboardOrder`/`listDashboardViews`, `/api/dashboard`, `/dashboard` page) is left fully intact and working. It has many callers (`structure-templates.ts`, `/build/surface`, `PinButton`), so its retirement + dropping `views.dashboard_order` moves to **Slice 2**, alongside the new react-grid-layout `/dashboards/[id]` page and the "add to dashboard" rewiring — keeping the build green throughout (a deviation from the plan's "retire in slice 1," for safety).
**Why / alternatives:** jsonb-per-dashboard over a normalized widget table — the exploration `dashboard-widgets.md` already anticipated "a `dashboard_layout` jsonb (per owner)… until multiple dashboards are real"; they now are, so the unit is one row per dashboard with widgets inline. react-grid-layout (the UI, Slice 2) is a user-approved Principle-5 exception (de-facto standard for resizable dashboard grids).
**Affects:** `src/db/schema.ts` (+`dashboards`), migration `0016_icy_dexter_bennett.sql`, new `src/lib/dashboards.ts`, new `src/app/api/dashboards/{route,[id]/route,reorder/route}.ts`, `src/app/api/items/query/route.ts` (+`count`), `src/lib/settings.ts` (+home/today ids), `src/lib/views.ts` (exported `parseSort`/`UUID_RE`/`VIEW_LIMIT` for reuse), new `scripts/verify-dashboards.mts` (19/19).

## ADR-066: External HTTP API — `/api/machine/items` (read out / write in) under a new `api` token scope
**Date:** 2026-06-15
**Status:** accepted (Tyler's lane). **Touches CORE — the machine/API contract** (CLAUDE.md "Building together"). Shipped solo on Tyler's call (the trigger: connecting Savor, his journalling app); **flagged to Brandon in `COLLAB.md` to ratify.** Additive only — no change to the existing MCP, cron, or webhook contracts.
**Context:** Until now the only token-authenticated read/write path was `POST /api/mcp` (JSON-RPC, built for AI clients). A plain app integration — e.g. Savor's cron pushing each new journal entry into Ledgr as an item — had no clean REST door: `/api/items*` is Clerk-login-only, and `/api/machine/*` was cron/admin endpoints, not item CRUD. MCP is an awkward fit for a simple "POST an item on a timer" job.
**Decision:**
1. **New endpoint `/api/machine/items`** (same door as the other `/api/machine/*` jobs — already in `proxy.ts`'s public set, Clerk-bypassed, the machine token IS the credential):
   - **`POST`** — create one item (a bare item object) or a **batch** (`{ items: [...] }`, max 100). The batch is the cron shape: push everything new since the last run. A malformed entry is reported in `errors` and **skipped, not fatal** (`{ count, created: Item[], errors: [{index, error}] }`; 201 if anything created, 400 if all failed). Every entry validates through the **same `parseItemPayload` / `createItem`** the in-app `POST /api/items` uses, so the external surface can't drift from the app contract or skip owner-scoping.
   - **`GET`** — owner-scoped, **body-free** list (the "read out" path); filters mirror the in-app list (`?type=&status=&parentId=&q=&limit=&offset=`).
2. **New `api` token scope** (the existing ADR-004 scheme; no token-format change). Generate with `node scripts/make-token.mjs <name> api`; `/health` is unchanged (no new canary this pass — the Build → AI & MCP page shows an `api`-token-present indicator instead).
3. **Owner resolution** via a new `resolveMachineOwner()` (`src/lib/machine/owner.ts`) — the single owner, by UPN, mirroring `resolveMcpOwner` but with its own optional knob `LEDGR_API_OWNER_UPN` (then the shared `ONEDRIVE_EXPORT_UPN`/`GRAPH_MAILBOX_UPN` fallbacks, then `DEV_USER_EMAIL` in dev). Kept separate from the MCP resolver so the REST surface isn't coupled to the MCP module; multi-user would map each token to its own owner.
4. **The item `type` is the integrator's choice.** The endpoint is type-agnostic — the caller sends a `type` key, so a journalling push lands in whatever type the owner sets up (a bespoke `journal` type in Build → Types, or a built-in like `note`). No type is created by this ADR.
**Why / alternatives:** A dedicated machine endpoint over (a) opening `/api/items*` to dual Clerk-or-token auth — keeps the app's Clerk routes simple and the machine surface explicitly scoped — and over (b) making integrators speak MCP — far simpler for a plain cron. Idempotency/dedup is **not** built v1: integrators push only new entries (cursor on their side); an optional `externalId` dedup is a noted follow-up. Surfaced in the UI under the existing Build → **AI & MCP** page (an "App integrations (HTTP API)" section), not a new nav entry.
**Affects:** new `src/app/api/machine/items/route.ts`, new `src/lib/machine/owner.ts`; `.env.example` (+`api` scope doc, +`LEDGR_API_OWNER_UPN`); `src/app/build/claude/page.tsx` (+HTTP API section). `proxy.ts` unchanged (`/api/machine(.*)` already public). **Pending: a `verify-machine-items.mts` (batch create, bad-entry-skipped, owner-scope, GET filters) + in-browser/curl smoke; flag Brandon to ratify the contract addition.**

## ADR-067: Typed relation properties (un-deferred), the `unmarked` placeholder type, and one create-on-miss rule
**Date:** 2026-06-16
**Status:** accepted. **Core** (the property model, a new system type, the relations write path, the create surfaces). Both-agree satisfied: Brandon designed it through this session, Tyler confirmed. Un-defers the `relation` property kind that ADR-044/ADR-055 and `schema.md` had parked; folds in the relation-kind note from `explorations/entity-vs-custom-type.md`. *(Numbered 067 because the dashboards build, ADR-064/065, and the external HTTP API, ADR-066, took those numbers first.)*
**Context:** Relations have shipped since slice 6/15: the generic edge table, the both-direction Related panel, the relate/un-relate/confirm write path (ADR-015), `@`-mention diff-sync (ADR-040), calendar suggested edges (slice 23), and template relation defaults (ADR-050). Two gaps remained. (1) There was no way to put a *typed* relation on a type, an "Author" box on a Book or an "Attendees" box on a Meeting that accepts a specific kind of thing; the `relation` property kind was written down but deferred because the generic `@`/+Relate surface already covered ad hoc links. (2) Create-on-miss: typing a name that does not exist yet should create it inline without leaving the item you are in (Brandon's "I should not have to navigate away mid-meeting"). Working it through clarified that Brandon's mental model, type the name into the Author box and it knows what to create, *is* the typed relation field, so the deferral no longer holds.
**Decision:**
1. **Un-defer the `relation` property kind. It is a property any type can declare, added by the user in the builder,** alongside text/number/date/checkbox/url/select/multi_select. A relation field carries, beyond the usual `{key, label, kind}`, a **target type** (which type its links accept, e.g. `person`; optional, unset means any type) and a **cardinality** (single or many).
2. **A relation field's value is stored as `relations` edges, not in `items.properties`.** The edge's `role` is the field's `key`, so the Author box reads as "edges from this item with role `author`" and writing to the box creates such an edge. This keeps relations in the relations table (the schema rule, and the ADR-050 precedent that relation defaults stay off the core property model) and means typed-field links appear in the generic Related panel for free. The existing unique `(source_id, target_id, role)` already lets several typed fields connect the same pair. Cardinality is enforced in the app layer (a single-relation field replaces its edge on a new pick), not by a DB constraint.
3. **One create-on-miss rule across every surface: if the box tells us the type, use it; otherwise the new item is `unmarked` and lands in the Inbox.**
   - **Inside a typed relation field** (the Author box): the type is known from the field, so create-on-miss makes that target type. No prompt.
   - **Free-text `@`-mention on the canvas, and the generic + Relate panel:** the type is unknown, so create-on-miss makes an `unmarked` item flagged `inbox = true`, links it, and keeps you in place. The earlier "person by default, switchable" idea is dropped in favor of this one uniform behavior (Brandon's simplification).
4. **A new seeded system type `unmarked` is the placeholder for "not yet typed."** It is `is_system = true` and **hidden** (ADR-059), so it never appears in quick capture, the +New menus, list tabs, or nav options. Its display **label is a glyph** (a small dash/dot; exact glyph is Brandon's pick), so the user never reads the word "unmarked"; the key `unmarked` is code-facing only. Triaging an unmarked item out of the Inbox is just a **retype** (already supported on the item PATCH path) plus clearing the inbox flag; the row, its links, and its body are untouched, so nothing is recreated.
5. **A literal null `items.type` is rejected.** Type drives the canvas, the export path (`{type}/{year}/…`), the FTS document, type grouping, and nearly every query, so a nullable type would turn all of those into permanent special cases for a state meant to be temporary. The `unmarked` type gives the same user-facing "typeless until I say" behavior with zero downstream special-casing, and Principle 6 already calls for a catch-all.
6. **Obsidian-style unresolved/lazy links are rejected.** An `@` to a non-existent thing could have left an unresolved reference that creates nothing until clicked and resolves by name later. Brandon does not want a wiki-style web of links and judged it added confusion and code for a feature he would not use. Every create-on-miss creates eagerly. The overwhelm worry (the Inbox filling with stubs) is accepted as the cost; bulk-triage mitigations can come later if it bites.
7. **The Related panel shows unmarked items** like any other related item, grouped under the glyph label, and **surfaces the role as a section heading** where a typed field set one (Attendees, References) so the optional labels show up where they exist.
8. **What this does *not* change.** Hierarchy stays the separate single-`parent_id` subtask/project tree, not an edge. There is no per-link classification: a plain link is just "connected," roles are optional labels, and the only place "classification" is declared is once per relation field (its target type and cardinality). The generic edge table, owner-scoping, soft-delete cascade, and the mention-edges-are-body-owned rule (ADR-015) are all unchanged.
**Why / alternatives:** *Keep the `relation` kind deferred* (status quo): rejected once Brandon's attendees/author boxes proved a real, recurring need the generic panel serves clumsily. *Store relation values in `items.properties` as target ids*: rejected because it splits link storage in two and breaks the Related-panel-shows-everything property; edges with a role keep one source of truth. *Nullable type* and *Obsidian lazy links*: rejected per points 5 and 6. *Person-default for the generic picker*: dropped for the simpler uniform `unmarked`-to-Inbox rule (point 3).
**Affects (when built, phased R1–R4 in `roadmap.md`/`next_steps.md`):** `src/lib/types.ts` (add `relation` to `PROPERTY_KINDS`; extend `PropertyDef` with `targetType`/`cardinality`; validate in `parsePropertySchema`); `src/components/build/TypeBuilder.tsx` + `CustomProperties.tsx` (author a relation field; render/edit it as a relation input over the relations API, not `items.properties`); `src/lib/relations.ts` (role-as-field-key reads/writes; a create-on-miss helper); `src/components/relations/` (`AddRelation`, the canvas `@` picker `mention-suggestion.ts`, a new typed-relation input) for create-on-miss; `scripts/seed.mjs` + a migration (seed the hidden `unmarked` type); `src/components/relations/RelatedPanel.tsx` (role section headings, unmarked group); `schema.md`, `CLAUDE.md`, `explorations/entity-vs-custom-type.md` (deferral lifted). No change to the `relations` table shape itself.

This is the first half of a two-feature item-view UX push (Brandon, 2026-06-16); the second half, a per-type drag-arrangeable item canvas layout, is **core** and gets its own ADR when built (see the COLLAB note and `next_steps.md`).

Clicking a related item already opens it as a modal (the intercepting route), so it was one click deep, not ten, but Brandon wanted the quick fixes to land without even that. ADR-063 already blesses editing structure from Work surfaces ("the separation is the default, not a wall... punch through it deliberately"); this builds the first concrete inline-edit surfaces for it.
**Decision:**
1. **A1 — edit a related item's title in place.** A related item's title (in the Related panel rows and the typed relation-field chips) renders with a small pencil off to one side, hover-revealed on desktop, that swaps the link for an input and PATCHes the item title (optimistic, revert on failure, `router.refresh()`). The title itself stays a click-through to the item, so anything deeper (its properties, body) is still one click away. Shared leaf `InlineTitle`.
2. **A2 — labels-only structure fixes, by right-click.** Right-clicking a type label or a property/relation-field label (the Related panel's type-group and field-section headings, the Properties/Relations panel field labels) opens a tiny menu: **Rename** (inline) or **Edit in builder…** (jumps to `/build/types/[key]/edit`). Rename moves only the display label; everything structural (keys, kinds, add/remove/reorder, target type) stays in the full builder. Shared leaf `InlineLabel`.
3. **Labels only, never the key.** Renaming routes through a new lightweight `PATCH /api/types/[key]/rename` (`{label}` for the type, `{label, propertyKey}` for a field) over new store fns `renameTypeLabel`/`renamePropertyLabel`. The key (the PK/FK, and a relation field's edge `role`) is immutable, so stored `items.properties` values and relation edges are untouched — `verify-inline-rename.mts` proves a typed edge survives a field-label rename. A dedicated endpoint (not the builder's whole-definition PATCH) means a one-label fix can't clobber a concurrent schema edit, mirroring the existing `setTypeQuickCapture` precedent.
4. **Mobile.** Inline editing is a desktop affordance: the pencil is hidden on touch widths and there's no right-click, so mobile falls back to tap-through (to the item, or to the builder). Long-press was rejected — it fights scrolling and text selection (Brandon).
5. **Ride-along: × to clear a property value** (`explorations/canvas-drag-and-drop.md` item 3). A filled scalar property shows a hover × that clears it to null. The uncontrolled text/url/number inputs get a remount key bumped on clear so the visible box empties too.
**Why / alternatives:** *Make the title itself editable (no pencil)* — rejected, it would steal the click-to-open gesture. *A modal-only edit* — already exists and is fine for deep edits; the pencil is the faster path for a one-field fix. *Full structure editing inline* — rejected for now (a slip would reshape the type for every item, and it clutters the reading surface); labels-only keeps the in-place edit safe, with the builder one menu-item away.
**Affects:** new `src/components/relations/InlineTitle.tsx`, `src/components/build/InlineLabel.tsx`, `src/app/api/types/[key]/rename/route.ts`, `scripts/verify-inline-rename.mts`; `src/lib/types.ts` (+`renameTypeLabel`/`renamePropertyLabel`, `parseLabel` extracted); `RelatedRow`/`RelationField`/`RelatedPanel`/`RelationProperties`/`CustomProperties` (wire the leaves, thread `typeKey`); `MarkdownCanvas`/`ChordCanvas` (pass `typeKey`). No migration, no `relations`/`items` shape change.

## ADR-069: Arrangeable, field-level, per-type item canvas layout (Feature B)
**Date:** 2026-06-17
**Status:** accepted. **Core** (a new `schema.md` column + the type/canvas model — both on CLAUDE.md's frozen core list). Both-agree satisfied: Brandon designed it (the 2026-06-16 session + the `explorations/item-canvas-layout.md` build brief); Tyler pre-blessed the direction (the COLLAB "Feature B WILL touch CORE" heads-up). The second half of the item-view UX push whose first half was ADR-068. Built straight through this session from the brief; verified in-browser against live Neon.
**Context:** The default item canvas (`MarkdownCanvas`) rendered a fixed vertical stack — title, a horizontal field strip, body, then the panels (subtasks/meeting-prep, custom properties, relations, Related, Save Offline, Share, the read-only Fields footer). PRD §4.13 wants "which fields go top-strip vs. footer, configurable per type." Brandon's ask went further: an item view should become "essentially zoneless, full DnD anywhere," arranged per type, with a sensible default so it's never forced. The dashboards already proved the engine (`react-grid-layout`, ADR-064); this brings the same 2D grid to the item canvas at field-level granularity.
**Decision:**
1. **One 2D grid, field-level cards.** The whole item window is a `react-grid-layout` grid; each field is its own draggable/resizable card — every system field (Status/Due/Urgency/When/URL), every custom scalar property, every relation field. The only cards not split: the markdown **body**, the **title**, and the **Related** backlinks panel (a dynamic list), each a single block. (Block-level "Properties panel" grouping was considered and rejected for field-level.)
2. **Per type, with a default; never forced.** Layout is stored per type in a new nullable **`types.canvas_layout` jsonb (migration 0019)**. `null` = the generated default → the **classic stacked render is untouched** (the common case, zero risk). Types are instance-global, so per-type = per-user here. The shape (`CanvasLayout = { version, cards: Record<CardId, {mode, hidden?}>, layouts: {lg, md, sm: Cell[]} }`) and all pure logic live in the client-safe **`src/lib/canvas-layout.ts`** (vocab/default/parse/reconcile/responsive), mirroring the `dashboard-widgets.ts ↔ dashboards.ts` split; `src/lib/types.ts` parses it tolerantly in `rowToDefinition` (bad shape → null, like a bad `property_schema`) and writes it via a focused `setTypeCanvasLayout` + **`PATCH /api/types/[key]/layout`** (mirrors the ADR-068 rename endpoint — a one-surface write that can't clobber a concurrent builder edit).
3. **Responsive, auto-derived, overridable.** Per-breakpoint layouts (`lg` 12-col / `md` 6-col / `sm` 1-col). The user arranges `lg`; `md`/`sm` are derived (`deriveResponsive`: sm = single column, md = lg re-packed into 6 cols) unless dragged at that width. `reconcile` runs on every read — drops cards no longer in the vocabulary (a deleted property), appends newly-added ones at the bottom — so the grid never goes stale when a type's schema changes.
4. **Body flow by default; pin-to-fixed optional.** A flow card's height follows its content; a fixed card is a set cell whose content scrolls. The make-or-break mechanic (a flow/auto-height body in the same grid as fixed cards) was de-risked first in a throwaway `/scratch/layout` spike: **measure the card's *natural* content height (never the RGL box) → rows, rAF-debounced, only re-set state when the row count changes** breaks the feedback loop; flow cards get width-only (`e`/`w`) resize handles. The spike graduated into `ItemRglInner` and was deleted.
5. **Three render paths.** `null` layout & not arranging → classic JSX (unchanged). Saved layout & not arranging → grid, read-only and chrome-free (no card headers, so a customized item still reads like a document). Arranging → grid + `.dash-edit` edit affordances + per-card pin/hide + a hidden-cards palette + auto-save + Reset-to-default. **Arrange entry is a full-page `?arrange=1`** reached by a hard `<a>` nav (escapes the intercept modal, the ADR-068 trick); the modal stays a quick reader (never arranges).
6. **RSC seam.** `MarkdownCanvas` (server) renders each card's content into a `Record<CardId, ReactNode>` and hands it to the client `ItemLayoutGrid` (`next/dynamic ssr:false` → `ItemRglInner`), exactly mirroring `DashboardGridLayout → RglInner`. The four async **server** panels (Subtasks, MeetingPrep, RelatedPanel, RelationProperties) are rendered server-side and passed through as nodes; the grid only positions them. The field panels are reused at field granularity by passing single-element arrays (`FieldStrip fields={[f]}`, `CustomProperties schema={[def]}`, `RelationProperties props={[def]}`); `ItemEditor` gained a `slot` prop (`title`/`body`/`full`) so title and body render as independent autosaving cards. A scoped `.canvas-card-content` CSS rule strips the panels' full-width canvas chrome (`mx-auto max-w-3xl px-12`) inside a cell.
7. **Atomic per-key property merge.** Because each scalar property is now its own card, a wholesale `{properties}` PATCH would let one card clobber another's value. Added **`propertyPatch`** to `ItemPatch`/`parseItemPayload` and `updateItem` (Postgres `coalesce(properties,'{}') || $patch::jsonb`, atomic, FTS recomputes) — merge only the changed key. `CustomProperties` now sends `propertyPatch`, which keeps the classic single-panel render correct too.
8. **Persistence discipline (a spike lesson).** A flow card's measured `h` is ephemeral — re-derived on load, never persisted. Layout writes happen only on real user actions while arranging (drag/resize/pin/hide/Done), gated like the dashboards' `if (!editMode)` rule, so a read-only view's reflow never writes back; RGL's `resizeHandles` artifact is stripped before persisting.
9. **Deferred.** Grouping/collapsible container cards (Brandon's "both/and") — the `CardMeta` shape leaves room (a future `group` with child ids + `collapsed`) without a rewrite. Bespoke module canvases (Chord/Paper) stay out of scope; Feature B is the default canvas only, and they opt into arrangeability at the module level later. PRD §4.13's "top-strip vs. footer per type" (the planned "B2") is subsumed — field-level placement delivers it; the `meta` card holds the read-only system footer.
**Why / alternatives:** *Block-level cards (whole panels)* — rejected for field-level, the finer grain Brandon wanted. *Persist the flow height / persist on every `onLayoutChange`* — rejected; the spike showed transient/derived layouts corrupt the stored shape (a staircased `md`), so flow `h` is ephemeral and writes are arrange-gated. *Wholesale `{properties}` PATCH per card* — rejected (clobber); `propertyPatch` jsonb-merge is atomic and keeps one code path for classic + cards. *Thread a `bare` prop through 7 panels* — rejected for a contained CSS neutralizer (the panels were authored for the full-width canvas; B1 favors the small reversible escape hatch). *Render card contents client-side from data (the dashboards' way)* — impossible here: four panels are async server components, so the node-map RSC-into-client pattern is required.
**Affects:** new `src/lib/canvas-layout.ts`, `src/components/canvas/ItemRglInner.tsx` + `ItemLayoutGrid.tsx`, `src/app/api/types/[key]/layout/route.ts`, `scripts/verify-canvas-layout.mts` (36 pure checks); migration **0019** (`types.canvas_layout`); `src/db/schema.ts` + `src/lib/types.ts` (thread `canvasLayout` + `setTypeCanvasLayout`); `src/components/canvas/MarkdownCanvas.tsx` (classic-vs-grid dispatch + node map + "Customize layout" entry), `ItemCanvas.tsx` + `src/app/items/[id]/page.tsx` (thread `arrange` from `?arrange=1`), `src/lib/modules.ts` (`CanvasProps.arrange?`); `src/components/markdown-editor/ItemEditor.tsx` (`slot` prop); `src/lib/items.ts` + `src/lib/item-input.ts` (`propertyPatch`); `src/components/build/CustomProperties.tsx` (send `propertyPatch`); `src/app/globals.css` (`.canvas-card-content`). Throwaway `/scratch/layout` spike added then deleted. `tsc` + eslint clean (new files); `verify-canvas-layout` 36/36, `verify-types`/`verify-items`/`verify-canvas-seam` green, `propertyPatch` merge round-trip verified on Neon; in-browser cycle (classic → arrange default → pin/hide → auto-save → Done grid → reload persisted → Reset → classic) confirmed against live Neon. No `relations`/`items` *shape* change beyond `propertyPatch` behavior.

**Post-build refinements (2026-06-17, Brandon feedback, all verified in-browser):**
1. **No repeated category labels.** The per-field card's panel heading ("Properties"/"Relations") is hidden via a `hideHeading` prop (CustomProperties/RelationProperties) when rendered as a single card, and the arrange card-header shows no label — just the ⠿ handle, the flow/fixed toggle, and ✕. The field's own name stays; the Related panel keeps its heading (Brandon: "I do like the label on related").
2. **One floating save indicator.** New app-wide singleton `src/lib/save-status.ts` (`beginSave`/`endSave`/`useSaveStatus`, a module-level pub/sub so it spans both the classic render and the grid where panels are scattered) + a fixed-position `SaveStatusIndicator` rendered once in `ItemCanvas`. ItemEditor's inline "Saved" badge was retired for it; ItemEditor/FieldStrip/CustomProperties report into it.
3. **Title + text properties wrap.** The title is an auto-grow wrapping `<textarea>` (Enter commits); text-kind custom properties are auto-grow wrapping textareas. Other kinds stay single-line; the explicit per-property wrap toggle was declined (default wrapping is enough — Brandon).
4. **Monochrome pin.** The flow/fixed control is a text toggle (neutral "Flow" / accent "Fixed"), not the colored 📌.
5. **Each surface maps to a breakpoint (Brandon's model).** The full-page expand fills the entire browser width → Desktop (`lg`); the modal panel (~768px) → Tablet (`md`); a phone → Phone (`sm`, one column). The grid fills its container (no max-width cap, `w-full`), so the breakpoint simply follows the surface, and "expand" is meaningfully bigger than the modal. A non-customized (classic) item keeps a readable column on the full page (`.canvas-wide` bumps its `max-w-3xl` blocks to ~64rem) rather than stretching prose full-width.
6. **Opening Arrange no longer commits a layout.** Persistence is gated to real gestures: `onLayoutChange` (which fires on mount + flow-measure) never persists; `onDragStop`/`onResizeStop` + pin/hide do. The Done button only flushes a *pending* change (timer-tracked) instead of force-saving, so peeking at Arrange and clicking Done leaves a `null`-layout type classic.
7. **Breakpoint thresholds + modal width.** `GRID_BREAKPOINT_PX = {lg:1024, md:480, sm:0}` (the md/sm boundary lowered from the dashboards' 768 so the only thing that collapses to one column is a true phone-width container; the modal's ~686px container is `md`/Tablet, not `sm`). The modal is NOT widened for a customized type — its default 3xl (768) width IS the Tablet surface. (An earlier pass mapped both modal and full page to `lg` and widened the modal to match; reverted in favor of the surface→breakpoint model above.) Verified: full page (1440) → 1360px Desktop, modal → 686px Tablet, phone (375) → 327px single column.
8. **Arrange every responsive layout from one editor — a Desktop/Tablet/Phone width switcher.** Previously only `lg` was arrangeable; `md`/`sm` auto-derived with no way to tune them. The full-page arrange toolbar has a width switcher that constrains the grid container to each surface's width (Desktop = full browser width, Tablet ≈ 700px to match the modal, Phone ≈ 420px), so RGL renders that breakpoint and the user can drag it (overriding the auto-derived `md`/`sm`). The switcher maps one-to-one to where the item is viewed: arranging "Tablet" customizes the modal, "Desktop" the expand. Per-breakpoint cells already persist independently; the measurer writes a card's flow height to the **active** breakpoint only (`onBreakpointChange` → `activeBp`), since heights differ by width. The grid **remounts on width change** (a `key`) so WidthProvider measures the newly-constrained container cleanly (a resize-nudge latched stale widths). Verified: Desktop 12-col, Tablet ~582px, Phone single column. Affects: new `src/lib/save-status.ts` + `SaveStatusIndicator.tsx`; edits to `ItemEditor`, `CustomProperties`, `RelationProperties`, `FieldStrip`, `ItemRglInner` (width switcher + per-breakpoint measure + key remount), `ItemCanvas`, `MarkdownCanvas`, `canvas-layout.ts` (breakpoints), `@modal/(.)items/[id]/page.tsx` (wide), `globals.css`.

## ADR-071: MCP tool additions — propertyPatch, relations, views, templates

## ADR-072: View + capture papercuts — catch-all capture default, list-filter by property, kanban drag-and-drop
**Date:** 2026-06-17
**Status:** accepted. **Non-core** (capture behavior + the views engine/UI, both on the "move fast, solo" side). Brandon-directed (his pick from the toward-1.0 papercut list); no migration. Flagged in `COLLAB.md` (shared view-engine + capture surfaces).

(`unmarked`'s glyph label `◌` stays for the Related panel; capture shows the friendly word.)

## ADR-073: Tasks go fully native — retire Todoist as the task + notification engine
**Date:** 2026-06-17
**Status:** accepted (Brandon — direction (B) of `explorations/tasks-todoist-vs-native.md`). **Core** (the `tasks` provider interface, the notification approach, and a new recurrence model are all cross-cutting). **Reverses PRD §4.6/§5.2** (which named Todoist the notification engine "by design"), in the spirit of ADR-037 — this ADR is the reversal record. Per-instance behind the seam: Brandon runs the native engine as the default `tasks` adapter; **Todoist stays an optional adapter** so Tyler (Google stack) can keep it. **Flagged to Tyler in `COLLAB.md` to ratify the core-seam change before the engine merges.**
**Context:** Today Ledgr is canonical for content and pushes dated tasks to Todoist (ADR-026), which owns recurrence, reminders, mobile capture, and the calendar feed — "the notification engine by design." The calendar time-blocking work (`explorations/calendar-time-blocking.md`) made Todoist *load-bearing for the calendar vision*, which is tail-wagging-the-dog: if Ledgr is where Brandon shapes his time, routing his own planned work through a third-party app to get it onto a calendar and to fire a reminder is a workaround, not the design. Brandon's decision: Ledgr owns tasks end to end.
**Decision:**

## ADR-076: Native task recurrence — model (C) per-date completion log, hand-rolled RRULE subset, a first-class `scheduled` column, virtual/materialized occurrences via one clone primitive
**Date:** 2026-06-17
**Status:** accepted (Brandon — the leanings in `explorations/recurrence-model.md` were his, settled here as the build record). **CORE** (the task data model + the native recurrence engine, the load-bearing T1 of the Native Tasks chunk, ADR-073). **Flagged to Tyler in `COLLAB.md` to ratify** the `tasks`-seam direction before the engine becomes the default adapter (ADR-073 point 7); the recurrence model itself is Brandon's call for his instance. Un-defers ADR-026's "recurrence delegated to Todoist." Resolves `explorations/recurrence-model.md`.
**Context:** Going fully native (ADR-073) means Ledgr owns recurrence, which Todoist owned entirely. T1 is load-bearing — it unblocks T2 (scheduling/overdue), T3 (focus surfaces the next occurrence), T4 (the ICS feed emits occurrences). The exploration laid out the model choice (A spawn-on-complete / B advance-the-date / C completion-log) and nine forks; this ADR settles each and records what was built.
**Decision:**
1. **Model (C), TaskNotes-style.** One `items` row per recurring task (the series) carries the rule + a per-date completion log; completing an occurrence stamps the date and advances to the next uncompleted occurrence. **No spawned clones, no overdue stacking** — missed dates are simply absent from the log. (A) was rejected (list fills with clones, missed dailies stack to N rows); (B) advance-the-date was rejected (no per-occurrence history). Everything stays an item (Principle 2).
2. **A first-class `scheduled_date` COLUMN** (migration 0021), distinct from `due_date` (the deadline). Chosen as a column, not a property, because it is *hot* — Today, the focus layer (T3), the ICS feed (T4), and the overdue auto-roll (T2) all query it — so it follows the schema rule "hot fields are columns," beside `due_date`, and stored the same UTC-midnight calendar-day way (ADR-008). An optional **`maintainDueOffset`** shifts `due` by the same delta when `scheduled` advances (the exploration's "maintain due offset"), default off.
3. **The rule + logs live in `items.properties.recurrence`** (jsonb): `{ rrule, dtstart, completeInstances[], skippedInstances[], occurrenceMode, anchorMode, maintainDueOffset? }`. Not a column — it isn't filtered per-row at scale, and "next occurrence" is *computed*, never stored as N rows (Principle 8). Tolerant-parsed (`parseRecurrence`, the views.ts discipline): a bad/absent rule reads as "not recurring."
4. **Hand-rolled, constrained RRULE subset — NOT the `rrule` npm package** (Principle 5). The rule is *stored* as a standards-shaped RFC-5545 RRULE string (`FREQ=DAILY|WEEKLY|MONTHLY|YEARLY`, `INTERVAL`, `BYDAY` for weekly, `COUNT`, `UNTIL`) so the ICS feed (T4) can emit it verbatim and a future swap to a full library is a drop-in; occurrences are *computed* by ~250 lines of deterministic code (Principle 3, no model). This matches the codebase's hand-roll precedents (ADR-034 web-push crypto, ADR-047 MCP, the no-date-lib `today.ts`) and keeps the dep count flat. The fiddly parts — DST, month-end (Jan 31 → Feb 28), leap years, BYDAY — are real, so the risk is bought down the ADR-034 way: a thorough verify with those exact cases. **`rrule` is the documented off-ramp** if the subset proves too small. Dates are calendar days (`YYYY-MM-DD`, UTC), which sidesteps DST entirely (a daily task never lands on the wrong day because clocks shifted); time-of-day reminder lead times layer on in T4.
5. **Occurrence mode, per task.** **virtual** (default): one series row + the log; completing it advances `scheduled` and the row stays `open` (the occurrence is done, the series isn't) until the rule ends (`COUNT`/`UNTIL`), when the row goes `done` and `scheduled` clears. **materialized**: each occurrence is its own item — a **deep clone of the series PROTOTYPE** (fresh unchecked subtasks, fresh body, carried tags/relations, `recurrence`/`occurrence` metadata stripped) — linked to the series by an **`occurrence` relation** (not `parent_id`, so the series' own canvas subtasks stay its only children, per ADR-061). Created via **create-next-after-completion**: completing an occurrence advances the series and clones the next, so **exactly one live occurrence exists at a time** and materialized never stacks either. **Subtasks or per-occurrence notes ⇒ materialized** (the rule of thumb); the engine auto-creates the first occurrence when a task is switched to materialized.
6. **One `cloneItemSubtree(resetRules)` primitive** (`src/lib/clone.ts`) does the deep clone-with-reset — the crux being each occurrence clones **from the pristine prototype, never from the just-completed (mutated) occurrence**, so checked subtasks/stale notes don't bleed forward. It is the shared primitive the exploration called for: the **templates system (ADR-045) can adopt it** to capture a subtree (the noted extension; templates seed body+defaults today, not subtasks).
7. **Fixed vs completion-based, per task** (`anchorMode`): **fixed** computes the next date from the RRULE calendar ("the 1st and 15th"); **completion** computes it from the completion date + interval ("3 days after I last did it" — Todoist's `every!`). Matters for chores/maintenance.
8. **Overdue display = next-only.** `scheduled` points at the single next uncompleted occurrence; missed dates don't pile up. A "N missed" badge is a later nicety (the helper exists).
9. **Edit series vs occurrence.** Editing the rule edits the series item (its `recurrence` property) and applies to future computation; editing one materialized occurrence edits that child (body/subtasks/notes) without touching the rule. RRULE-splitting for "this and future" is a deliberate v1 scope cut.
10. **Skip** (`skippedInstances`, distinct from done and missed) and **end conditions** (`COUNT`/`UNTIL`) are first-class in the rule.
11. **Completion is recurrence-aware everywhere.** `updateItem` intercepts the open→done transition, so the canvas checkbox, the MCP `update_item` tool, and the REST API all advance recurrence identically — no caller needs to know the rules.
**UI (T1):** `scheduled` joins the task canvas field strip (and footer), and a **`RecurrenceControl`** sets the rule (repeat preset + interval + weekday picker + on-schedule/after-completion + one-task/separate-note + the live description), writing `properties.recurrence` via the atomic `propertyPatch`. Wired into both canvas paths — the classic stacked render and the Feature-B grid (a new flow `recurrence` card + `sys:scheduledDate`). Reschedule shortcuts + natural-language dates are T2.
**Why / alternatives:** Spawn-on-complete (A) and advance-the-date (B) rejected (see point 1). `rrule` package rejected on Principle 5 (point 4; off-ramp documented). `scheduled` as a property rejected — it's a hot queried field (point 2). Occurrence-as-`parent_id`-child rejected — it conflates the series' canvas subtasks with occurrence instances; an `occurrence` relation keeps them separate (point 5). Materialized-on-every-cycle-in-place (delete+recreate the series' subtasks each cycle) rejected — cloning a history item is the same work and preserves "what did I decide last month."
**Verification:** `scripts/verify-recurrence.mts` — **60+ checks**, all PASS. Pure engine (no DB): calendar math incl. DST-boundary, month-end clamp, leap-year; RRULE parse/format round-trip; enumeration for daily/weekly/BYDAY/biweekly/monthly/yearly with COUNT/UNTIL; next-uncompleted skipping logged dates; complete/skip transitions (fixed: advance + no-stacking + series-end; completion: completedOn+interval); tolerant parse. **Service against live Neon** under throwaway owners: virtual completion advances + stays open + logs; counted series → done + scheduled cleared; maintain-due-offset; `cloneItemSubtree` (fresh unchecked subtasks, body copied, relation carried); materialized create-next-after-completion (one live occurrence, no stacking, series log advanced); owner scoping. Plus an **in-browser end-to-end over the real HTTP API** (create → set weekly recurrence via `propertyPatch` → complete → advanced to next Wednesday, stayed open, date logged). `tsc` + full `next build` + eslint clean; `verify-canvas-layout` (updated for the new cards) + `verify-canvas-seam` + `verify-items` green. Migration 0021 applied to Neon (additive nullable column, no reseed needed).
**Affects:** new `src/lib/recurrence.ts` (pure engine), `src/lib/recurrence-service.ts` (completion/materialization), `src/lib/clone.ts` (the shared primitive), `src/components/canvas/RecurrenceControl.tsx`, `scripts/verify-recurrence.mts`; `src/db/schema.ts` (+`items.scheduled_date` + index), `drizzle/0021_*`; edited `src/lib/items.ts` (scheduledDate + recurrence-aware completion), `src/lib/item-input.ts` (parse scheduledDate), `src/lib/canvas-fields.ts` + `src/components/canvas/FieldStrip.tsx` (Scheduled field), `src/lib/canvas-layout.ts` (recurrence + scheduled cards), `src/components/canvas/MarkdownCanvas.tsx` (mount, both paths), `scripts/verify-canvas-layout.mts`; `schema.md`. No new dependency. Resolves `explorations/recurrence-model.md`.

## ADR-077: Native task scheduling — hand-rolled NL dates, reschedule shortcuts, a deterministic overdue auto-roll, future-scheduled = deferred
**Date:** 2026-06-17
**Status:** accepted (Brandon). T2 of the Native Tasks chunk (ADR-073, point 4). Builds on the `scheduled_date` column + recurrence engine from ADR-076. Non-core (a date parser, a scheduling helper, and UI over the existing item APIs — no schema change beyond what T1 added).
**Decision:**
1. **Natural-language dates, hand-rolled** (`src/lib/nl-date.ts`) — no `chrono-node`/date library (Principle 5; same call as the recurrence RRULE subset). A constrained, predictable grammar (Principle 3, not an NLP model): `today`/`tomorrow`/`tonight`/`yesterday`, `in N days|weeks|months` (and bare `N days`), `next week|month`, weekday names with `next` = +1 week, `weekend`, `jun 20` / `june 20 2026` / `20 jun`, `m/d[/yy]`, and ISO passthrough; a no-explicit-year date already past rolls to next year; an unrecognized phrase returns null (the field is left for the user, never guessed). Pure + client-safe (caller passes the reference today), reusing the recurrence date helpers. Wired into the scheduled field and the capture modal.
2. **Reschedule shortcuts on the scheduled field** — Today / Tomorrow / +1wk chips beside the date picker + the NL input, rendered for the task's `scheduled` field in both canvas paths (classic + the Feature-B grid `sys:scheduledDate` card). Optimistic PATCH, the FieldStrip pattern.
3. **Deterministic overdue auto-roll** (`src/lib/scheduling.ts`) — rolls every open, live, **non-recurring** task past its **effective plan date** (scheduled if set, else the due deadline) forward to `scheduled = today`, **leaving the deadline alone** (a missed due date stays a fact). Recurring series are excluded — their progression is completion-driven (ADR-076), so auto-advancing would silently skip a logged occurrence. Surfaced as a one-click "Roll N overdue → today" on Today (shown only when N > 0, counting just what would move), `POST /api/tasks/roll-overdue` (manual), and an optional cron `GET /api/machine/roll-overdue` (the ADR-036 machine-token door; not scheduled by default).
4. **Defer/start = a future `scheduled` date.** Today now partitions tasks on the effective plan date, and only surfaces tasks planned/due by today — so a task scheduled for a future day stays off Today until then (the defer behavior, no new field). A scheduled-but-undue task shows a "planned" marker so its date isn't read as a deadline.
**Why / alternatives:** A date-parsing dependency rejected (Principle 5; a planner types a handful of shapes). Rolling the *due* date too (Todoist's reschedule) rejected — `scheduled` is the plan, `due` the deadline; the roll moves the plan and the missed deadline stays visible. A separate `defer`/`start` column rejected — a future `scheduled` already means "not yet," so Today filtering delivers defer for free. Auto-roll on a forced daily cron rejected as default — the one-click button is the deterministic, user-invoked path; the machine endpoint is there for whoever wants to schedule it.
**Verification:** `scripts/verify-scheduling.mts` — 26 pure NL-date checks (every grammar shape + rejection + year-rollover) + live-Neon overdue-roll (only open non-recurring effective-overdue tasks move to today; recurring/future/done left alone; due-only-overdue gains scheduled=today and keeps its deadline; count matches; owner scoping) — ALL PASS. In-browser: the "Roll 2 overdue → today" button rolled + vanished; the Tomorrow chip set + persisted scheduled. `tsc` + full `next build` + eslint clean.
**Affects:** new `src/lib/nl-date.ts`, `src/lib/scheduling.ts`, `src/components/today/RollOverdueButton.tsx`, `src/app/api/tasks/roll-overdue/route.ts`, `src/app/api/machine/roll-overdue/route.ts`, `scripts/verify-scheduling.mts`; edited `src/components/canvas/FieldStrip.tsx` (reschedule shortcuts + NL on scheduled, `today` prop), `src/components/canvas/MarkdownCanvas.tsx` (pass `today`), `src/components/capture/CaptureModal.tsx` (NL due), `src/lib/today.ts` (effective-plan-date partition + scheduled on Today), `src/app/page.tsx` (roll button + partition). No new dependency.

## ADR-080: Offline capture — a localStorage outbox flushed on reconnect
**Date:** 2026-06-17
**Status:** accepted (Brandon). T5 of the Native Tasks chunk (ADR-073, point 6). Non-core (a client-side queue + capture wiring; no schema, no API change). Replaces the Todoist inbox pull-in (ADR-010/026) as the offline-capture path.
**Decision:**
1. **An offline outbox** (`src/lib/outbox.ts`) — when a quick capture can't reach the server (offline or a transient network failure), the payload is queued locally and flushed on reconnect, so capture never loses a thought without a third-party app. **localStorage, not IndexedDB** — capture payloads are tiny (a title + a few fields), single user, so a synchronous JSON array is plenty and far simpler (Principle 5).
2. **Capture surfaces enqueue on failure instead of erroring.** Quick Capture (Today) and the capture modal `catch` a failed POST → `enqueueCapture(payload)` + a "Saved offline · will sync" affordance (Quick Capture) / silent close (modal). ## ADR-083: Recurrence completions calendar + the inverted occurrence edit (carve-out)
**Date:** 2026-06-18
**Status:** accepted (Brandon; **Tyler greenlit the whole Tasks Polish chunk** up front). S3 of the Tasks Polish chunk. **Not core** — it adds app-level operations over the *existing* recurrence model (ADR-076): no schema change, no change to the `properties.recurrence` shape, no change to the machine/MCP contract or the `tasks` seam. Builds directly on ADR-076 (the per-date completion log) and replaces ADR-076 §9's deferred "this and future" RRULE-split with a cleaner carve-out. **Flagged to Tyler in `COLLAB.md`** as a heads-up (it's in the recurrence area; his module-type tasks inherit the calendar for free).
**Context:** ADR-076 shipped the recurrence engine: a virtual series carries an RRULE + a per-date completion log, and the done-checkbox advances `scheduled` to the next uncompleted occurrence. Two gaps remained for daily use: (1) you could only complete *the current* occurrence (the checkbox), never tick a specific past/future date or fix a mistake out of order; (2) ADR-076 §9 cut "edit this occurrence" to a v1 scope-out. S3 closes both, exactly as Brandon settled 2026-06-18.
**Decision:**
1. **Completions = a month-calendar on the recurring task** (TaskNotes "Completions" style). A `RecurrenceCalendar` renders the RRULE projection for the viewed month (occurrence days are buttons, non-occurrence days inert); **every occurrence date is individually checkable in any order**, toggling its entry in the per-date completion log (`properties.recurrence.completeInstances`). The "up next" occurrence is highlighted, completed days filled with a ✓, carved/skipped days struck through.
2. **A toggle is a direct log edit, NOT the checkbox gesture.** New pure helpers in `recurrence.ts` (`isOccurrence`, `toggleCompleteInstance`, `addSkippedInstance`, `instanceState`) mutate only the log — they compute no next date. The service (`toggleOccurrenceCompletion` in `recurrence-service.ts`) applies the toggle then **recomputes `scheduled` = the next uncompleted occurrence on/after *today*** (forward-looking — `nextUncompletedOnOrAfter(rule, today)`), with status following (done + scheduled cleared when nothing remains forward, else not-started). So missed *past* dates stay absent from the log and never resurrect as `scheduled` (ADR-076 §1/§8), but ticking today's occurrence advances and un-ticking it returns. Distinct from `completeOccurrence`/`skipOccurrence`, which branch on the anchor mode for the "I just did the current one" gesture.
3. **The inverted occurrence edit (replaces "this and future").** Editing a recurring task edits the **series** by default. To shape one date differently, **carve it out**: `carveOccurrence` deep-clones the series PROTOTYPE (the shared `cloneItemSubtree`, ADR-076 §6 — fresh unchecked subtasks, fresh body, carried relations, **no** recurrence) into a fresh **DETACHED** one-off scheduled on that date, the series adds the date to `skippedInstances` and advances past it, and the user lands on the new item to edit it. **The carve is NOT linked by the `occurrence` role** (so the materialized create-next-after-completion machinery never fires on it) — completing the carved one-off leaves the series untouched. "Edits never touch the parent," verified.
4. **Virtual-series only.** A materialized series' occurrences are already their own items with their own checkboxes; editing the series log there would desync. The canvas shows the calendar only for `occurrenceMode === "virtual"`, and `toggleOccurrenceCompletion`/`carveOccurrence` guard it (bad_request otherwise).
5. **One endpoint, two actions:** `POST /api/items/[id]/occurrence` with `{action:"toggle"|"carve", date}` (the date must be a valid occurrence; the route validates the YMD). Both recompute the series' scheduled + status server-side, so the client only edits the log. The calendar is a new card (`recurrenceCalendar`, label "Completions") in the field-level canvas vocabulary (ADR-069), placed right after the Repeat control; it's `null` for a non-recurring (or materialized) task, which `ItemRglInner`'s `visibleIds` filter already drops cleanly.
**Why / alternatives:** *A "this and future" RRULE split* (ADR-076 §9) rejected for the carve — splitting a series into two rules is fiddly and rarely what's meant; carving one date into a detached copy is simpler and matches the mental model ("just this one is different"). *Linking the carve by the `occurrence` role* (like a materialized occurrence) rejected — it would trip create-next-after-completion and re-couple the one-off to the series, the opposite of "detached." *Anchoring the scheduled recompute at `dtstart`* (earliest uncompleted ever) rejected — a long-running daily series would resurrect months of untouched past dates as `scheduled`; today-anchored keeps misses absent (ADR-076 §1). *Editing the log via the generic `propertyPatch` from the client* rejected — the server must recompute `scheduled`/status atomically from the edited log, so a dedicated endpoint owns the rule. *Carving on a materialized series* rejected — its occurrences are already separate items. *`window.confirm` for the carve* — banned (ADR-057); an inline confirm bar in the calendar (one UI, two triggers: the per-cell ✎ and the "Edit next occurrence" button) is the in-context standard.
**Verification:** `scripts/verify-recurrence-calendar.mts` — pure (`isOccurrence` incl. COUNT/UNTIL bounds, `toggleCompleteInstance` add/remove/sort/dedupe + clears-skipped, `addSkippedInstance`, `instanceState`) + live-Neon service (toggle future = logged + scheduled unchanged; toggle today = scheduled skips to next uncompleted; untoggle returns scheduled; non-occurrence rejected; counted series → done + scheduled cleared; carve = detached one-off scheduled on the date with no recurrence + no occurrence edge + a fresh subtask, series skips + advances, completing the one-off leaves the series put; materialized/non-recurring/owner-scope guards) — **ALL PASS (39 checks)**. Regression: `verify-recurrence` green; `verify-canvas-layout` updated (vocab + completions card) and green. **In-browser against live Neon:** built a weekly-Wed series, the calendar rendered its Wednesdays; ticking Jun 17 marked it ✓ and advanced `scheduled` to Jun 24 (persisted across reload); "Edit next occurrence (Jun 24)" → confirm → carved a detached one-off (scheduled Jun 24, no recurrence, fresh "Prep agenda" subtask) and the series skipped Jun 24 + advanced to Jul 1, navigating to the new item. `tsc` + full `next build` + eslint clean. No migration.
**Affects:** new `src/components/canvas/RecurrenceCalendar.tsx`, `src/app/api/items/[id]/occurrence/route.ts`, `scripts/verify-recurrence-calendar.mts`; edited `src/lib/recurrence.ts` (+`isOccurrence`/`toggleCompleteInstance`/`addSkippedInstance`/`instanceState`/`InstanceState`), `src/lib/recurrence-service.ts` (+`toggleOccurrenceCompletion`/`carveOccurrence`/`writeSeriesLogState`), `src/lib/canvas-layout.ts` (+`recurrenceCalendar` card kind/spec/vocab/label), `src/components/canvas/MarkdownCanvas.tsx` (parse the rule once; mount the calendar in both canvas paths, virtual-only), `scripts/verify-canvas-layout.mts` (vocab assertion). ## ADR-085: Relative subtask scheduling — a stored offset from the parent's scheduled date, set by calendar-pick
**Date:** 2026-06-18
**Status:** accepted (Brandon; **Tyler greenlit the whole Tasks Polish chunk** up front). S5 of the Tasks Polish chunk. **Not core** — a scheduling behavior over existing data (a new `properties.relativeSchedule` shape, no column/migration), plus additive hooks in two core primitives (`updateItem`, `cloneItemSubtree`). **Flagged to Tyler in `COLLAB.md`** (it touches clone.ts + updateItem, which his materialized occurrences + templates ride). This is the "short ADR (subtask date semantics)" S5 called for.
**Context:** A recurring/scheduled task's subtasks usually want to land *relative* to the parent — "send invites 2 days after the event date," "prep the day before." Hard-coding each subtask's absolute date breaks the moment the parent moves or a recurring occurrence lands on a new date. Brandon's call (2026-06-18): store an offset, set it by picking a date (Ledgr back-calculates the offset), recompute per occurrence.
**Decision:**
1. **The offset is the source of truth; `scheduled_date` is derived.** A subtask carries `properties.relativeSchedule = { offsetDays: N }` (jsonb, no column — light owner data like `properties.recurrence`/`focus`). Its `scheduled_date` **column** holds the concrete derived day (parent's scheduled + N), so every existing scheduled query/sort/Today/ICS/overdue-roll keeps working unchanged. N may be negative ("2 days before"). Pure helpers in client-safe **`src/lib/relative-subtask.ts`** (`parseRelativeSchedule`/`relativeOffsetOf`/`offsetBetween`/`applyOffset`/`describeOffset`); calendar-day UTC math (ADR-008).
2. **Set by calendar-pick; Ledgr back-calculates.** The `SubtaskSchedule` control on each task subtask row: pick a date → if the parent has a scheduled date, store `offsetDays = picked − parentScheduled` (relative); with no parent date, store a plain absolute date (no offset). The row shows the date + a `+Nd`/`−Nd` chip.
3. **Recompute triggers (deterministic, no model):** (a) **parent scheduled-date change** — `updateItem` calls `recomputeRelativeChildren` (`src/lib/relative-subtask-service.ts`) whenever a task's `scheduledDate` changes, re-deriving every relative child and **chaining down** (a relative child's own relative children re-derive against the child's new date); (b) the **recurrence advance / S3 calendar toggle/carve** paths (`advanceSeriesRow`/`writeSeriesLogState`) call it too, since they set `scheduled_date` outside `updateItem`; (c) **occurrence materialize** — `cloneItemSubtree` derives a relative child's date from the new clone's scheduled day + the carried offset (the offset isn't stripped), so a materialized occurrence's subtasks land N days out. **Offsets are relative to the IMMEDIATE parent** (they chain), matching "the parent's scheduled date."
4. **No anchor ⇒ cleared, not stale.** If the parent loses its scheduled date, a relative child's derived date clears (the offset stays, so it re-derives when the parent is re-dated). An **absolute** (non-relative) subtask is never touched by a parent move.
**Why / alternatives:** *A `scheduled_offset` column* rejected — it's light owner data and only relative subtasks have it (Principle 2/8: jsonb like recurrence/focus); the hot column stays the concrete `scheduled_date`. *Storing only the offset and computing the date on read* rejected — every scheduled query would need a per-row parent lookup; deriving the concrete date on write keeps reads an indexed scan. *Offset relative to the series root (not the immediate parent)* rejected — "the parent's scheduled date" means the immediate parent, and chaining composes naturally. *Recomputing through `updateItem` (re-entrant)* rejected — `recomputeRelativeChildren` writes children directly and recurses with a depth cap, avoiding the recurrence/mention/category side-effects of a full update. *An absolute-vs-relative toggle in the UI* deferred — picking a date with a dated parent is relative by default (the feature); a future toggle can add a "pin to an absolute date" escape hatch.
**Verification:** `scripts/verify-relative-subtasks.mts` — pure (parse tolerance, `offsetBetween`/`applyOffset` incl. negative + month-cross, `describeOffset`) + live-Neon (relative child shifts with the parent, absolute child untouched, clears when the parent loses its date + re-derives when re-dated with the offset preserved, offsets chain parent→child→grandchild, and the clone/materialize path derives a relative child's date while a plain child stays undated) — **ALL PASS (26 checks)**. Regression: `verify-recurrence`/`verify-recurrence-calendar`/`verify-subtasks` green. **In-browser against live Neon:** on a parent scheduled Jun 25, the subtask's "＋ when" → picking Jun 27 stored `relativeSchedule {offsetDays:2}` + scheduled Jun 27; rescheduling the parent to Jul 2 shifted the subtask to Jul 4 (offset preserved); the row chip read "Jul 4 +2d". `tsc` + full `next build` + eslint clean. No migration.
**Affects:** new `src/lib/relative-subtask.ts` (pure), `src/lib/relative-subtask-service.ts` (recompute), `src/components/subtasks/SubtaskSchedule.tsx`, `scripts/verify-relative-subtasks.mts`; edited `src/lib/items.ts` (recompute on scheduled change), `src/lib/recurrence-service.ts` (recompute after advance / calendar log write), `src/lib/clone.ts` (derive relative child dates on clone), `src/lib/subtasks.ts` (+`relativeOffset` on `SubtaskNode`, select `properties`), `src/components/subtasks/Subtasks.tsx` (thread `parentScheduled`, render the control), `src/components/canvas/MarkdownCanvas.tsx` (pass `parentScheduled`). No migration, no new dependency.

## ADR-087: Meeting recording v1a — the `transcript` child type, the Transcript panel + awaiting-minutes view, and the Principle-3 reading for a Claude-over-MCP minutes automation
**Date:** 2026-06-18
**Status:** accepted (Brandon; **Tyler pre-approved the feature and all three core touches** 2026-06-18, see `explorations/meeting-recording.md` "Core flags" + the COLLAB approval). This is **v1a** of the meeting-recording chunk (paste-first; no audio pipeline yet — that's v1b). The data/UI pieces (the `transcript` type, the panel, the view) are **non-core**; **the Principle-3 interpretation in point 5 is the CORE element this ADR ratifies.** **Flagged to Tyler in `COLLAB.md`.**
**Context:** Meeting recording → transcription → AI minutes is a committed 1.0 item. The converged design (`explorations/meeting-recording.md`, 2026-06-18) made the **transcript the pivot** (audio is disposable, deferred to v1b), upload/paste-only, privacy not a blocker, and — the key move — **processing is a Claude-over-MCP workflow, not an in-app LLM call.** v1a proves the valuable part (transcript → minutes + tasks) with almost no new infrastructure: no outbound LLM call, no transcription service, no storage concern.
**Decision:**
1. **A transcript is its own item, not a region of the meeting body.** A new `transcript` system type (migration 0023 + `seed.mjs`; `is_system`, visible, `show_in_quick_capture=false`). Rationale: a 20k–35k-word transcript would swamp the meeting body editor and the human-facing doc; a separate item also lets a meeting carry several transcripts. It is **not** added to `coreModule` — an unregistered seeded type falls back to the default markdown canvas (the same path `paper`/`song` would take if unregistered), so no core module edit.
2. **A transcript is tied to its meeting two ways, deliberately.** `parent_id = the meeting` (containment — the subtask soft-delete cascade carries it to Trash) **and** a confirmed `relations` edge `meeting→transcript`, role `transcript` (association). The edge is what makes the connection first-class to the MCP graph: `get_item(meeting)` lists its transcripts in `related`, and `list_items(type=transcript, relatedTo=<meeting>)` finds them — so "look at recent meetings, pull their transcripts, explore" is one hop, not a fragile title search (Brandon's explicit concern). `createTranscript` (`src/lib/meetings/transcripts.ts`) writes both in one place so a transcript can't exist invisible to the automation.
3. **The "needs minutes" signal is a `select` property on the transcript** (`minutes`: `none`/`draft`/`done`), seeded in the type's `property_schema`; new transcripts are stamped `minutes=none` at create so the index-backed `properties @>` filter matches. A saved view **"Transcripts awaiting minutes"** (`type=transcript`, `propertyFilters:[{minutes:none}]`, seeded per owner by `scripts/seed-meeting-views.mts`) is the automation's work queue. `none` → processed → `draft` → reviewed → `done`; only `none` is "awaiting," so the automation never double-processes.
4. **Minutes land in the meeting body; suggested tasks land in the Inbox related to the meeting** (Brandon's leanings). Tasks are real `task` items (`inbox=true`, `relateTo` the meeting), not `unmarked` — they're unambiguously action items, so forcing a retype would be friction. (Trivially changed; it's a line in the prompt.)
5. **CORE — the Principle-3 reading (this ADR amends how Principle 3 is applied).** Principle 3 says "AI lives only in the deliberate Claude/MCP layer, never in a cron job." A *scheduled* Claude-over-MCP automation brushes against the "never in a cron" clause. The ratified reading: **the principle keeps Ledgr's own backend model-free and deterministic** (no Anthropic key in the app, no model in a Ledgr cron, no summary seam) — and an **external** Claude automation hitting the public MCP/API and producing **staged** outputs (minutes as a draft you edit, tasks in the Inbox for triage, the transcript left `draft` not `done`) **honors that intent**. The safeguard that keeps it clean is that **nothing auto-commits**; the human-in-the-loop staging is load-bearing, not incidental. The processing prompt is documented (`docs/meeting-minutes-automation.md`) and surfaced on Build → AI & MCP; it satisfies "tweak the prompt" for free (it's text Brandon edits) and generalizes as the blueprint for every future "AI on purpose" feature. If Brandon ever wants it fully hands-off (auto-commit), that's a further decision.
**Why / alternatives:** *Transcript as a `## Transcript` region in the meeting body* rejected — swamps the editor + the doc, and forecloses multiple transcripts. *minutes-state on the meeting* rejected in favor of the transcript — "Transcripts awaiting minutes" is then a clean single-type view, and the transcript is the unit the automation processes. *An in-app `generate_minutes` MCP tool / outbound LLM call* deferred (v1b "Deferred" list) — the existing 12 tools (ADR-047/071) already express the whole flow (`run_view`→`get_item`→`update_item`→`create_item`→`update_item` with `propertyPatch`), so v1a needs no new tool and no first outbound LLM call. *A typed `relation` property "Transcripts" on the meeting type* (ADR-067) rejected for the surface — the bespoke Transcript panel is the richer surface; a plain role-`transcript` edge gives the graph/Related-panel discoverability without a redundant field box.
**Verification:** `scripts/verify-meeting-transcripts.mts` **16/16 on live Neon** — create writes the child + the confirmed `transcript` edge + `minutes=none`; the MCP-discovery path (`get_item(meeting).related` and `list_items(relatedTo)` both surface the transcript); word counts; the awaiting-minutes filter (none in, draft/done out); guards (parent must be the owner's live meeting). `scripts/verify-canvas-layout.mts` still green (new `meetingTranscripts` card in the meeting vocabulary). `tsc` + full `next build` + eslint clean. **In-browser on live Neon:** opened a meeting → Transcript panel; pasted a transcript → appeared as "No minutes yet · N words"; opened the transcript → full editor + the Minutes dropdown (none/draft/done); the awaiting view listed it; setting `minutes=done` via the canvas (the `propertyPatch` write the automation uses) dropped it from the view. Test data cleaned up.
**Affects:** migration `drizzle/0023_meeting_transcript_type.sql` (+ journal/snapshot) + `scripts/seed.mjs` (the `transcript` type); new `src/lib/meetings/transcripts.ts`, `src/app/api/meetings/[id]/transcripts/route.ts`, `src/components/meetings/MeetingTranscripts.tsx` + `AddTranscript.tsx`; edited `src/components/canvas/MarkdownCanvas.tsx` (the panel, classic + grid card) and `src/lib/canvas-layout.ts` (the `meetingTranscripts` card); new `scripts/seed-meeting-views.mts` (the awaiting view), `scripts/verify-meeting-transcripts.mts`, `docs/meeting-minutes-automation.md`; edited `src/app/build/claude/page.tsx` (the Automations section). **v1b (the `transcription` provider seam + AssemblyAI adapter, audio upload, the `attachments.purge_after` retention column + purge) is deferred to its own ADRs (the other two pre-approved core touches).**

## ADR-088: The `transcription` provider seam + an AssemblyAI adapter (meeting recording v1b)
**Date:** 2026-06-18
**Status:** accepted (Brandon; **Tyler pre-approved — the `transcription` seam is one of the three pre-approved core touches**, `explorations/meeting-recording.md` "Core flags"). **CORE** (provider interfaces are core), additive, mirrors the existing seams. **Flagged to Tyler in `COLLAB.md`.** This is **V1b-1** of the meeting-recording chunk; audio upload/auto-transcribe (V1b-2) and audio retention (V1b-3, ADR-089) build on it.
**Context:** v1a (ADR-087) made the transcript the pivot and shipped the paste path. v1b adds *convenience* transcription: upload audio, Ledgr transcribes it into a transcript. Per CLAUDE.md provider-interface discipline (storage/auth/scheduler/calendar/mail/push/tasks), the transcription API must sit behind a thin seam so the adapter can be swapped and a future local build is a packaging exercise. Transcription is a deterministic, user-triggered API call — fine under Principle 3 (the model here is a speech-to-text service, not a reasoning step; only the *minutes* are "AI on purpose," and those stay in the Claude-over-MCP layer, ADR-087).
**Decision:**
1. **A `transcription` provider seam** (`src/lib/transcription/{types,provider,assemblyai}.ts`), env-selected, **null-safe** (returns `null` when unconfigured, exactly like `getStorage()`), so the paste path and the whole app work without it — audio transcription is simply not offered. The interface is a **submit + poll job model** (`submit(audioUrl, opts) → {jobId}`, `poll(jobId) → TranscriptionResult`), because long meetings transcribe over minutes and a serverless request can't block that long; the caller submits on upload and polls from a status check/cron. `TranscriptionResult` carries `status` (queued/processing/completed/error), `text`, diarized `segments` (`{speaker, start, end, text}`), and `error`. A pure `transcriptToMarkdown` renders a finished transcript (diarized → "**Speaker A:** …" paragraphs; plain otherwise).
2. **AssemblyAI is the first adapter** (Brandon's lean) — real bundled speaker diarization, takes long files by URL with no chunking, clean REST. **No SDK dependency** (Principle 5): two `fetch` calls (POST `/v2/transcript` with `audio_url` + `speaker_labels`, GET `/v2/transcript/{id}`), and the response→result mapping is a **pure `mapTranscriptResponse`** (tolerant: unknown status → "processing" so polling continues), node-testable with fixtures and no live key.
3. **Enabled by the presence of `ASSEMBLYAI_API_KEY`** (the deliberate opt-in, like the R2 vars enabling storage); `TRANSCRIPTION_ADAPTER` selects the adapter (default "assemblyai") and can be set to "none" to disable even with a key. `transcriptionAdapter()` is the pure id (mirrors `tasksAdapter()`), surfaced as **`/health` `checks.transcription`**.
**Why / alternatives:** *A blocking `transcribe()` that submits + waits* rejected — a 3-hour file exceeds any serverless timeout; the job model is the only shape that fits. *OpenAI gpt-4o-transcribe* rejected (the spec's cost table) — a hard 25MB/25-min cap forces chunking on long meetings; AssemblyAI/Deepgram/Gemini take long files directly. *Gemini one-call transcribe+summarize* deferred — weaker prompt-based diarization, and it would fold the summary back into the API instead of the Claude-over-MCP layer; the seam keeps it swappable if that changes. *An SDK* rejected (Principle 5) — two signed-less REST calls don't earn a dependency.
**Verification:** `scripts/verify-transcription.mts` (pure, no DB/key) **ALL PASS** — `mapTranscriptResponse` over queued/processing/completed-with-diarization/error/unknown-status fixtures; `transcriptToMarkdown` diarized vs plain; the null-safe env-selected selection (unconfigured → none/null; key → assemblyai; explicit none disables). `verify-health-check`/`verify-provider-seams` green (the new `checks.transcription` canary). `tsc` + eslint clean. **Live submit/poll against AssemblyAI is gated on a configured key + a real audio URL — pending (the graph-auth integration-gated posture); it gets its in-browser eyeball when V1b-2 lands the upload path with a key set.**
**Affects:** new `src/lib/transcription/types.ts`, `assemblyai.ts`, `provider.ts`; edited `src/lib/health.ts` (`checks.transcription`); new `scripts/verify-transcription.mts`; edited `scripts/verify-health-check.mts` (fixture) + `.env.example` (`TRANSCRIPTION_ADAPTER`/`ASSEMBLYAI_API_KEY`). No migration, no new dependency. **V1b-2 (audio upload → submit on completion → poll → fill the transcript) and V1b-3/ADR-089 (the `attachments.purge_after` retention column + purge) build on this seam.**
